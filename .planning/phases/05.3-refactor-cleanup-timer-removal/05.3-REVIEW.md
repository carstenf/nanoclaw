### WR-01: AmdClassifier.onAudioDelta is dead wiring â€” Â§201-StGB audio-leak detection never fires

**File:** `voice-bridge/src/amd-classifier.ts:114-122, 276-289`
**File:** `voice-bridge/src/sideband.ts:295-315` (trace-only handler)
**Issue:** The classifier interface declares `onAudioDelta(bytes: number)` and
`isAudioLeaked(): boolean`, with the load-bearing invariant (header comment
lines 14-17): "Â§201 StGB zero-audio-leak: no bot audio before voicemail
verdict. If response.audio.delta arrives before verdict, log warn + set
audioLeaked." Inside `createAmdClassifier`, `onAudioDelta` sets the
`audioLeaked` flag on line 279-288.

However, sideband.ts never forwards `response.audio.delta` events to the
classifier. The only place `response.audio.delta` is observed is the Spike-A
trace path (`maybeWriteTrace`, sideband.ts:296), which redacts bytes for
Â§201 logging but does NOT invoke `classifier.onAudioDelta`. Result:
`isAudioLeaked()` always returns `false`, the `case_2_audio_leaked_before_verdict`
warn log (amd-classifier.ts:282-287) can never fire in production, and the
invariant documented in the header is silently broken.

This is not a legality-breach by itself (audio is still not persisted), but
the missing wiring means the observability the invariant promises does not
exist â€” a regression in Case-2 AMD hygiene that would be detected by this
log would go unnoticed.

**Fix:** In `sideband.ts` message handler, add a forward to the active
classifier (and surface via `SidebandOpenOpts.onAudioDelta` matching
`onSpeechStart/onSpeechStop`):

```ts
// In openSidebandSession onmessage, after maybeWriteTrace:
if (parsed?.type === 'response.audio.delta') {
  const bytes = typeof parsed.delta === 'string'
    ? Buffer.byteLength(parsed.delta, 'base64')
    : 0
  opts.onAudioDelta?.(bytes)
  return
}
```

and in `call-router.ts` wire it through:

```ts
onAudioDelta: (bytes: number) => {
  getAmdClassifier()?.onAudioDelta(bytes)
},
```

Alternative: if this observability is intentionally unused post-Spike-A,
delete `onAudioDelta`/`isAudioLeaked` from the interface and the dead
implementation to stop the header comment from lying.

### WR-02: Module-level _activeClassifier leaks on human-verdict path and on normal endCall

**File:** `voice-bridge/src/tools/dispatch.ts:44-47`
**File:** `voice-bridge/src/webhook.ts:425-501` (onHuman closure)
**File:** `voice-bridge/src/call-router.ts:184-237` (endCall)
**Issue:** `setAmdClassifier(classifier)` is called at webhook.ts:517 for
every Case-2 outbound /accept. The module-level reference is only cleared
in two places:

1. `buildCase2OnVoicemailHandler` sets it to `null` on voicemail verdict
   (webhook.ts:178).
2. Tests explicitly call `setAmdClassifier(null)`.

It is NOT cleared when:

- The `onHuman` callback runs (webhook.ts:425-501) â€” the classifier stays
  registered until the call ends. If the model emits a spurious second
  `amd_result` after the human verdict (possible with gpt-realtime per
  05.2-06 VAD-fallback notes), `dispatch.ts:191` will route it to the
  already-settled classifier. `settleHuman`/`settleVoicemail` guard with
  `verdict !== 'pending'`, so the double-emit is a no-op â€” but the leaked
  reference still points at an old call's classifier when the next
  non-Case-2 inbound arrives.
- `call-router.endCall` runs. `_activeClassifier` is only replaced on the
  next Case-2 call via `setAmdClassifier(classifier)` at webhook.ts:517.
  Between a Case-2 call ending (voicemail-path clears it, human-path does
  not) and the next Case-2 call arriving, the module-level state can hold
  a stale reference from a prior call.

Worst-case blast radius is small (all state is guarded by `verdict` and
`stopped` flags, and a Case-6b inbound call will land in the `!classifier`
branch of dispatch.ts:174 because the reference IS set on the prior
Case-2 call â€” wait, that's the issue: a stale classifier makes the
`amd_result` path look active on a non-Case-2 session, which then routes
the amd_result to a classifier for a wrong call-id). In practice the
model won't emit amd_result without the classifier prompt, so this is
latent.

**Fix:** Add `setAmdClassifier(null)` to the `onHuman` closure after the
persona swap, and/or clear it from `call-router.endCall` for Case-2 calls.
Simpler: make cleanup unconditional at endCall so the module never holds
a cross-call reference:

```ts
// In call-router.ts endCall, before map.delete(callId):
import { getAmdClassifier, setAmdClassifier } from './tools/dispatch.js'
// ...
const classifier = getAmdClassifier()
if (classifier) {
  classifier.stop()
  setAmdClassifier(null)
}
```

Note the cross-call coupling is itself a design smell (module-level
mutable state in a single-process multi-call server). A follow-up would
move `_activeClassifier` into `CallContext`, but that is refactor scope
beyond this finding.

### WR-03: Number() coercion on user-supplied casePayload fields silently produces NaN

**File:** `voice-bridge/src/webhook.ts:431-432`
**File:** `voice-bridge/src/outbound-router.ts:223` (`partySize`)
**Issue:** On the AMD-human-verdict path (webhook.ts:431):

```ts
time_tolerance_min: Number(casePayload.time_tolerance_min ?? 30),
party_size: Number(casePayload.party_size ?? 1),
```

If `case_payload` carries `time_tolerance_min: ''` or `time_tolerance_min: 'null'`
(strings), `Number()` returns `NaN`, which is then passed into
`buildCase2OutboundPersona` and rendered into the persona text as the string
`"NaN"` via template interpolation. Similar issue in outbound-router.ts:223
where `partySize = Number(casePayload.party_size ?? 1)` produces `NaN`, and
the subsequent `partySize !== 1` evaluates true (so "Personen" is appended)
â€” the notification text ends up "fĂĽr NaN Personen".

No injection risk (the values are cast through `String()`/numeric anyway),
but the user-facing TTS/notification output is broken for malformed Core
input. The Core outbound-webhook zod schema (outbound-webhook.ts:45)
accepts `case_payload: z.record(z.string(), z.unknown()).optional()` â€” so
the shape is not validated at the bridge boundary.

**Fix:** Use a numeric guard so defaults kick in on non-numeric input too:

```ts
function numOr(v: unknown, d: number): number {
  const n = typeof v === 'number' ? v : Number(v)
  return Number.isFinite(n) ? n : d
}
// ...
time_tolerance_min: numOr(casePayload.time_tolerance_min, 30),
party_size: numOr(casePayload.party_size, 1),
```

Apply consistently in `reportBackCase2` (outbound-router.ts:223).

### WR-04: Unsanitized user-supplied restaurant_name flows into voice_notify_user text

**File:** `voice-bridge/src/webhook.ts:167-169`
**File:** `voice-bridge/src/outbound-router.ts:226, 237, 265, 272`
**Issue:** In `buildCase2OnVoicemailHandler` (webhook.ts:167):

```ts
text: `Voicemail erkannt bei ${String(casePayload.restaurant_name ?? 'Restaurant')} (${reason}). NĂ¤chster Versuch in KĂĽrze.`,
```

`casePayload.restaurant_name` is user-supplied (Core enqueue payload), flows
unchanged into the `voice_notify_user` text. `sanitizeForPersona` (strip `{}`)
is applied only for persona-instructions text (persona.ts:72, overlay
consumption). Notification text is NOT persona-interpreted, but can contain
arbitrary bytes â€” newlines, control characters, large multi-KB payloads, or
fragments that a downstream Discord renderer would mistreat.

Same pattern in `reportBackCase2` (outbound-router.ts:226, 237, 265, 272)
where `restaurantName` is built from `casePayload.restaurant_name` at line 206
WITHOUT sanitization â€” only webhook.ts:167 uses `String()`, and
outbound-router.ts:206 uses `String()` as well. Neither strips control chars
or enforces length caps.

Severity is Warning (not Critical) because: (a) `voice_notify_user` already
has its own zod schema in Core, (b) Discord rendering is not a trust
boundary for the bridge, (c) the restaurant_name must pass outbound-webhook
zod validation before arriving here (though `case_payload: z.record` is
permissive).

**Fix:** Apply `sanitizeForPersona` OR a stricter `sanitizeForNotification`
(strip `{}` + newlines + clamp to e.g. 200 chars) at the boundary where
notification text is built:

```ts
const restaurantName = sanitizeForPersona(
  String(casePayload.restaurant_name ?? 'Restaurant'),
).slice(0, 200)
```

Even better: push the sanitization one level up â€” have
outbound-webhook.ts Zod-validate a `restaurant_name: z.string().min(1).max(200)`
inside `case_payload` (instead of the generic `z.record`), so the bridge
never sees unbounded strings.

### WR-05: getPeerIp throws on malformed empty-array x-forwarded-for header

**File:** `voice-bridge/src/outbound-webhook.ts:54-64`
**Issue:**

```ts
const xff = request.headers['x-forwarded-for']
if (xff) {
  const first = (Array.isArray(xff) ? xff[0] : xff).split(',')[0].trim()
  // ...
}
```

If `xff` is `[]` (falsy, so `if (xff)` catches), fine. But if `xff` is `['']`
(single empty-string element) or `[undefined as any]`, `xff[0]` is `''` or
`undefined`. For `undefined`, `.split(',')` throws `TypeError: Cannot read
properties of undefined (reading 'split')`. Fastify normalizes headers to
strings under normal circumstances, but pathological proxies (or tests that
inject `headers: {'x-forwarded-for': []}`) can trigger this.

The outer `try/catch` in the Fastify request pipeline would catch and
return 500, but it's a reachable crash path that bypasses the normal
`allowlist.has(peerIp)` check.

**Fix:**

```ts
function getPeerIp(request: {
  headers: Record<string, string | string[] | undefined>
  ip: string
}): string {
  const xff = request.headers['x-forwarded-for']
  const xffStr = Array.isArray(xff) ? xff[0] : xff
  if (typeof xffStr === 'string' && xffStr.length > 0) {
    const first = xffStr.split(',')[0]?.trim()
    if (first) return first
  }
  return request.ip
}
```

### IN-06: SlowBrainWorker.stop() abandons the runLoop without awaiting it

**File:** `voice-bridge/src/slow-brain.ts:183-186`
**Issue:** `stop()` flips `running = false` and aborts the current RPC, but
does not `await` the `runLoop()` promise (it's fire-and-forget on line
168: `void runLoop()`). An in-flight `coreClient.callTool` will still
complete after `stop()` returns, and its result (including any
`updateInstructions`) will fire against a possibly-closed sideband.
`updateInstructions` guards with `state.ready && state.ws`, so the effect
is a no-op WARN log â€” benign but noisy.

**Fix:** Store the runLoop promise and `await` it in `stop()`:

```ts
const loopPromise = runLoop()
// ...
async stop(): Promise<void> {
  running = false
  currentAbort?.abort()
  try { await loopPromise } catch { /* swallow */ }
},
```

This also avoids an "unhandled promise rejection" if runLoop ever throws
unexpectedly.
