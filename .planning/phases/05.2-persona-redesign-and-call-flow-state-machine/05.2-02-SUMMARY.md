# Phase 05.2 Plan 02: Silence-Monitor State-Machine Rewire (D-7) Summary

Bot-audio-aware silence monitor: "Bist du noch da, Carsten?" no longer fires mid-bot-turn. Timer is armed only when BOTH the bot and caller are silent, cancelled whenever either starts speaking.

## What landed

### silence-monitor.ts — state-machine rewire
Extended `SilenceMonitor` interface with `onBotStart()` and `onBotStop()`. Factory closure now tracks two booleans:
- `botSpeaking` — set `true` on `onBotStart()`, `false` on `onBotStop()`
- `callerSpeaking` — set `true` on `onSpeechStart()`, `false` on `onSpeechStop()`

Arming rule (research §4.3 verbatim): timer is armed only when BOTH flags become `false`. Specifically:
- `onSpeechStop` arms only if `!botSpeaking` (otherwise defers to later `onBotStop`)
- `onBotStop` arms only if `!callerSpeaking` (otherwise defers to later `onSpeechStop`)
- `onBotStart` / `onSpeechStart` cancel any armed timer unconditionally

Inline code-comment block cites the OpenAI Realtime Server Events reference URL — `output_audio_buffer.stopped` is the conservative "bot truly finished" signal (fires after `response.done`), research §4.2.

### sideband.ts — event plumbing
Added two new cases to the WS `onmessage` switch, placed immediately after the existing `input_audio_buffer.speech_started/stopped` cases:
- `output_audio_buffer.started` → `opts.onBotStart?.()`
- `output_audio_buffer.stopped` → `opts.onBotStop?.()`

New optional fields `onBotStart` / `onBotStop` on `SidebandOpenOpts`, with JSDoc citing research §4.3.

### call-router.ts — call-scope wiring
Added `onBotStart` / `onBotStop` handlers to the `openSidebandSession` options, routing WS events to the per-call `silence.onBotStart()` / `silence.onBotStop()` methods — mirrors the existing `onSpeechStart` / `onSpeechStop` wiring. Without this, sideband.ts would fire events into a no-op and the bug would persist (Rule 3 auto-fix — blocking issue, documented in Deviations below).
