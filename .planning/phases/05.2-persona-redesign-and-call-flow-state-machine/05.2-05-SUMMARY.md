## Decisions Made

**Q7 verdict — INCONCLUSIVE, docs-lean ATOMIC**

Documentation research leans strongly toward ATOMIC per:
- OpenAI Cookbook "Realtime Prompting Guide — Dynamic Conversation Flow via session.updates" shows `set_conversation_state` replacing prompt+tools in a single session.update with no special handling (no `response.cancel`, no two-step, no wait). If non-atomic, this would be a documented race condition; the Cookbook does not flag it
- OpenAI Realtime Server Events reference silent on atomicity (no published guarantee, but also no documented race)
- Zero community forum reports of instructions/tools split-visibility

Empirical probe authored but not executed (no API key in parallel-executor sandbox). Probe is idempotent; re-running it will overwrite the finding file with ATOMIC or NON-ATOMIC verdict. Test E branches on verdict at test-setup time, so re-running the probe automatically updates test expectations.

**Task 3 Branch A selected (no functional change)**

The post-AMD webhook.ts onHuman handoff pushes `instructions` only. `updateInstructions()` at sideband.ts:704-710 actively strips any `tools` field (D-26/AC-05 invariant — enforced with BUG-level logging). The Case-2 tool list (13 tools including `amd_result`) was fixed at `/accept` via `openai.realtime.calls.accept(...)` and is not re-pushed in the onHuman closure. This means Q7 atomicity of session.update({instructions, tools}) **cannot** affect the 05.2 handoff — the handoff never pushes both fields together. Q7 relevance is deferred to Phase 5 state-graph transitions that may push both.

**Case-2 onHuman does NOT arm armedForFirstSpeech (D-8 narrowing preserved)**

webhook.ts:594-600 already implements the correct split: only the non-Case-2 (Case-1 default-outbound) branch sets `state.armedForFirstSpeech = true`. The Case-2 onHuman closure uses the explicit setTimeout → requestResponse chain from Plan 05.1-01 Layer-2. Arming the flag in Case-2 would cause a double response.create (once from first counterpart speech_stopped, once from the explicit setTimeout). Already correct — no code change needed. Verified via Test B (no response.create in pre-verdict WS sends).
