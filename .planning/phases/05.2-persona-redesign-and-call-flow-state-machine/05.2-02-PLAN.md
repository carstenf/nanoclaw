Fix the silence-monitor state-machine bug (D-7, research §4): current `silence-monitor.ts:155-159` arms the countdown on caller `speech_stopped` with NO awareness of whether the bot is currently speaking, causing "Bist du noch da, Carsten?" to fire immediately after the bot finishes its own sentence (live defect observed 2026-04-21, described verbatim in research §1.3 item 2 and §4.1).

Wave 1, parallel with 05.2-01 (baseline persona) — zero file overlap.

Root cause (research §4.1): the timer counts from an earlier caller `speech_stopped` event that may have fired WHILE the bot was still talking — VAD on the input buffer is indifferent to output buffer state. Carsten observed: "kaum hat er seinen satz beendet, kommt schon 'carsten bist du noch da'".

Fix (research §4.2-§4.3): arm countdown on `output_audio_buffer.stopped` (bot-just-finished — this is the conservative "bot definitely finished speaking" signal per OpenAI Realtime server-events reference) OR on `input_audio_buffer.speech_stopped` (caller-just-finished). Cancel on `output_audio_buffer.started` (bot speaking again) OR `input_audio_buffer.speech_started` (caller speaking). Never armed at construction time.

D-7 scope: STATE-MACHINE change only, NOT a persona-prompt change. Persona-level silence rules (the TRANSCRIBED round1/round2/round3 text that the model is told to say) stay unchanged — those remain in legacy persona.ts CASE6B_PERSONA/PHASE2_PERSONA (and in baseline.ts from 05.2-01 for new cases). ### Target silence-monitor.ts API extension

```typescript
export interface SilenceMonitor {
  onSpeechStart: () => void       // caller VAD — EXISTING: cancels timer
  onSpeechStop: () => void        // caller VAD — EXISTING: arms timer
  onBotStart: () => void          // NEW: bot audio buffer started — cancels timer
  onBotStop: () => void           // NEW: bot audio buffer stopped — arms timer
  stop: () => void
  _round: () => number
}
```

### State-machine behavior (research §4.3 verbatim — paraphrased for code)

```
States (implicit — captured as boolean flags `botSpeaking` and `callerSpeaking`):
  IDLE          — both flags false, no timer armed
  BOT_SPEAKING  — botSpeaking=true, timer cancelled
  CALLER_SPEAKING — callerSpeaking=true, timer cancelled
  WAITING_FOR_CALLER — both false, timer armed (silenceMs countdown)

Event → action:
  output_audio_buffer.started → onBotStart() → botSpeaking=true, clear() timer
  output_audio_buffer.stopped → onBotStop()  → botSpeaking=false, IF !callerSpeaking schedule()
  input_audio_buffer.speech_started → onSpeechStart() → callerSpeaking=true, clear() timer, reset round counter
  input_audio_buffer.speech_stopped → onSpeechStop()  → callerSpeaking=false, IF !botSpeaking schedule()

Key invariant (§4.3): the timer is NEVER armed while either flag is true. Only armed at the moment BOTH become false.
```
