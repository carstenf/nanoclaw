Core MCP tool zod schema (voice_notify_user):

```typescript
export const VoiceNotifyUserSchema = z.object({
  call_id: z.string().optional(),
  turn_id: z.string().optional(),
  text: z.string().min(1).max(4000),
  urgency: z.enum(['info', 'decision', 'alert']),
  target_jid: z.string().optional(),  // override — default = main group JID
});
```

Return shape (exact):

```typescript
type VoiceNotifyUserResult =
  | { ok: true; result: { routed_via: 'whatsapp' | 'discord'; delivered: true } }
  | { ok: false; error: 'no_main_group' | 'routing_failed' | 'long_text_but_no_discord' | 'internal' };
```

Routing decision (documented in voice-notify-user.ts — this is authoritative):

```
1. Count words in `text` (whitespace-split; > 0 length). Constant = VOICE_NOTIFY_LONG_TEXT_WORD_THRESHOLD = 50.
2. If word_count > 50 → force route = 'discord'. Reason logged: 'long_text_override'.
3. Else: consult getActiveChannelFor(target_jid, now):
     - returns 'whatsapp' if any WhatsApp channel reports isConnected() AND the active-session-tracker
       has seen inbound activity on a WhatsApp-owned JID within VOICE_ACTIVE_SESSION_WINDOW_MS
       (default 600000ms = 10 min) of `now`.
     - returns 'discord' if a Discord channel reports isConnected() AND no recent WhatsApp activity
       OR WhatsApp is not connected at all.
     - returns null if neither channel has any activity within the window AND neither is connected.
4. If the resolver returns null → attempt Discord fallback; if Discord delivery fails → log alert + return `{ ok: false, error: 'routing_failed' }`.
5. On any routing choice, append JSONL event `voice_notify_user_routed` with { urgency, word_count, routed_via, delivered, reason?, latency_ms }.
```

active-session-tracker contract:

```typescript
export interface ActiveSessionTracker {
  /** Called by Andy's message ingest with the channel name + JID + timestamp_ms of the inbound message. */
  recordActivity(channel: 'whatsapp' | 'discord', jid: string, ts_ms: number): void;
  /** Looks up which channel Carsten most-recently sent a message on for a given JID within window_ms. */
  getActiveChannelFor(jid: string, now_ms: number): 'whatsapp' | 'discord' | null;
  /** DI test hook — exposes the internal map size. */
  _size(): number;
}
```

The tracker is intentionally NOT wired into the main message loop by this plan — that wiring is Wave 2's
job (see Plan 05-02 Task 5). For Wave 1, the tool ACCEPTS a DI tracker so tests can run with a fake tracker.
The default factory returns a tracker that reads from a simple in-memory map; nothing wires it to
inbound-message events yet. This is an interface-first task ordering discipline (per planner rules).
