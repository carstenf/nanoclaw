<domain>
## Phase Boundary

Close the architectural drift between the Bridge ↔ NanoClaw-Core tool-dispatch spec intent and the shipped implementation. The spec (ConOps + PRD AC-07/AC-09 + ARCHITECTURE-DECISION.md + REQUIREMENTS DIR-04/DIR-10/C6B-03) calls for **MCP in both directions** on this path. What exists today:

- **Production path (Bridge → Core):** `voice-bridge/src/core-mcp-client.ts` is a JSON-POST REST facade (`POST {url}/{tool_name}` with `{ arguments }` body), served by `src/mcp-server.ts` on port 3200. No JSON-RPC envelope, no `initialize` handshake, no capabilities negotiation, no discovery — despite the file name.
- **Spec-compliant path (Chat-Claude/iOS debug):** `src/mcp-stream-server.ts` on port 3201, stateless StreamableHTTP, added Phase 4 Wave 3. Shares the same `ToolRegistry` as port 3200.

Phase 4.5 retires the REST facade, migrates the bridge onto the spec-compliant StreamableHTTP channel, resolves the iOS-client-hang (now on the production path, not a debug-only channel), and re-aligns REQUIREMENTS.md wording.

**In scope:**
1. New MCP-SDK StreamableHTTP client inside `voice-bridge/src/core-mcp-client.ts` with long-lived session (one session per live voice call, reused across many tool calls).
2. Server-side `src/mcp-stream-server.ts` switched from stateless (per-request server/transport) to session-based (`sessionIdGenerator` enabled, session Map with TTL, Last-Event-ID resume).
3. Proper tool metadata (description + `inputSchema`) for all tools currently exposed without schema — both because MCP clients need them and because without schemas the iOS app's behavior is undefined.
4. Port 3200 REST server (`src/mcp-server.ts`) enters deprecation window — kept running in Phase 4.5, removal scheduled in a follow-up phase after bridge migration has baked.
5. Voice-bridge cost gate (`voice-bridge/src/cost/gate.ts`) and pre-greet (`voice-bridge/src/pre-greet.ts`) migrate their tool calls onto the new MCP client.
6. Regression harness — synthetic "initialize + tools/list + tools/call × N" client covering both the per-request hang mode (would have caught the 2026-04-19 bug) and the session-based flow. Lives in `src/mcp-stream-server.test.ts` or a new `tests/mcp-regression.test.ts`.
7. Doc updates: `REQUIREMENTS.md` AC-07 wording, `/opt/server-docs/hetzner-mcp-architecture.md` "Bekannte Probleme" entry, decision file `.planning/decisions/2026-04-20-mcp-unblock.md` iff tools are renamed/removed during investigation.

**Out of scope (explicitly):**
- Outbound call path Sipgate REST (REQ-SIP-02 — external boundary, stays).
- Bridge `/outbound` HTTP trigger surface (REQ-INFRA-13 — internal-REST by design, stays).
- Tool-surface expansion beyond Phase 4's 18 tools.
- Port 3200 server deletion — deprecate here, remove in a follow-up.
- Phase 4's cost-ledger / drift-monitor / recon jobs — already shipped.

<decisions>
## Implementation Decisions

### Protocol & Transport

**D-1. Bridge ↔ Core = MCP StreamableHTTP (single protocol both directions)**
The bridge MUST use `@modelcontextprotocol/sdk`'s StreamableHTTP client. The JSON-POST REST facade in `core-mcp-client.ts` is replaced in-place (same module path, same exports where possible so downstream imports don't churn). Source: ARCHITECTURE-DECISION.md "Bridge ↔ NanoClaw = MCP in both directions", PRD AC-09, REQ-DIR-10.

**D-2. Server-side transport is session-based, NOT stateless**
`StreamableHTTPServerTransport` initialized with `sessionIdGenerator: () => randomUUID()`. The current per-request server/transport factory in `src/mcp-stream-server.ts:232-277` is replaced with a session Map (`sessionId → { server, transport, lastActivity }`), TTL sweep (default 30 min idle), reconnect support via `Last-Event-ID`. Rationale: stateless mode produces iOS-client UI-hang (2026-04-19 evidence). Session-based is also required so the bridge can hold one MCP session per live voice call and amortize the handshake across N tool calls.

**D-3. Session TTL and cleanup policy**
- Idle TTL: 30 minutes default (configurable via `MCP_STREAM_SESSION_TTL_MS`).
- Active TTL: unbounded while the client holds the session (a live voice call can run 10+ min).
- Sweep interval: 60 seconds, `unref()`'d so it doesn't block process exit.
- On sweep, sessions past TTL: `transport.close()` + `server.close()` + remove from Map.
- Memory budget: expect ≤ 10 concurrent sessions (1 bridge + a few iOS debug clients). Per-session cost is one `McpServer` instance + one transport — empirically ~300KB. Cap hard at 50 sessions; reject new `initialize` beyond that with a 503.

**D-4. Tool metadata completeness**
Every tool registered via `server.tool()` gets a non-generic description and a `zod` input schema matching its real handler contract. Current state (`src/mcp-stream-server.ts:153-199`): only 4 of 18 tools have `TOOL_META` entries. The other 14 fall back to `server.tool(name, genericDescription, handler)` — the SDK emits an empty `{ type: "object" }` schema. This is spec-legal but loses semantics for MCP clients. All 18 tools get proper descriptions + `inputSchema`.

**D-5. Tool name format: keep `voice.` dot-namespace**
Dots are spec-legal in MCP tool names (the spec constrains nothing beyond string). The 2026-04-19 hang is reproduced on paths that don't touch tool names (initialize → tools/list was green), so dots are not the root cause. Renaming all 18 across bridge + iOS + tests + docs is ~200 LOC of churn for speculative benefit — **NOT done** in 4.5. Tracked as `decisions/backlog/` if a future client truly rejects dots.

**D-6. Bridge client — long-lived session per call**
`voice-bridge/src/core-mcp-client.ts` opens one MCP session when a voice call starts (on first tool need, lazy init) and holds it until `session.closed`. Teardown: `client.close()` in the call-session finalizer. This matches the Realtime session lifecycle — one `initialize` handshake pays off over ~10-50 tool calls per 5-minute conversation.

**D-7. Connection pooling — NOT now**
One session per call. No pool, no warm session. If call setup latency becomes an issue (first tool call has to wait for handshake), revisit with measurement in Phase 5+. Premature optimization.

### Deprecation & Migration

**D-8. Port 3200 REST server stays running during 4.5**
`src/mcp-server.ts` continues to bind 3200 — but nothing in the bridge or any other consumer points at it after 4.5 lands. `listNames()` audit log + per-request tool-name counter on both ports lets us verify zero traffic before removal. Removal phase is NOT 4.5. Tracked as follow-up in ROADMAP.md.

**D-9. Migration order (safe, reversible)**
1. Server: add session-based transport alongside stateless. Run tests against both.
2. Server: switch stateless → session-based, remove the stateless code path. Deploy, verify iOS unblock.
3. Bridge: introduce new MCP client module `voice-bridge/src/core-mcp-client-v2.ts` alongside v1. Unit tests.
4. Bridge: switch `slow-brain.ts` + `pre-greet.ts` + `cost/gate.ts` to v2. Leave v1 module importable for one release.
5. Bridge: delete v1 module, rename v2 → `core-mcp-client.ts`. Follow-up commit.
6. Observability: log `mcp_rest_request_seen` on any hit to `src/mcp-server.ts` routes. If 0 over a quiet week, green-light port 3200 removal in a follow-up phase.

**D-10. Backward compatibility during the migration window**
Port 3200 and 3201 both live. Tool registry is shared (single source of truth for tool behavior). Any tool added to the registry is reachable via both ports during the window. No divergent tool logic, ever.

### Error Handling & Guardrails

**D-11. Pitfall 8 (synthetic call_id/turn_id) stays enforced**
Chat-Claude / iOS invocations must continue to receive synthetic `call_id = chat-<uuid>` / `turn_id = chat-<ts>-<slice>` so they can never collide with live voice-call idempotency keys. Currently done in `mcp-stream-server.ts:205-213`. Move this into the session setup (`.tool()` handler wrapper) so it applies regardless of which session a chat client connects from.

**D-12. Auth layering unchanged**
Bearer (MCP_STREAM_BEARER) + WG-peer allowlist on port 3201. Same allowlist as port 3200 (`10.0.0.1, 10.0.0.2, 10.0.0.4, 10.0.0.5`). Bridge on 10.0.0.2 will talk to 10.0.0.2:3201 locally — allowlist already covers this. iOS connects via Caddy which strips the OAuth token and forwards with the fixed bearer.

**D-13. OAuth/Caddy layer unchanged**
`/nanoclaw-voice/*` Caddy route + OAuth forward_auth + RFC 8414/9728 discovery endpoints landed 2026-04-19 and are green. Nothing in 4.5 touches Caddy or OAuth. If session-mode requires the `Mcp-Session-Id` header to be forwarded (which it does), verify Caddy passes it through by default — add explicit `header_up` if not.

**D-14. `voice.on_transcript_turn` stays on tools/list**
The briefing's path-1 hypothesis (iOS auto-invokes any tool named `*_transcript_*`) is unverified and speculative. We don't remove a tool from the registry just to quiet an iOS bug; the tool is bridge-internal and has legitimate callers. If the iOS unblock fails after D-2 (session mode), revisit removal — but only with evidence (server-side log showing iOS sent an uninvited `tools/call` for it).

### Verification & Regression

**D-15. Synthetic regression harness — mandatory**
New test file (exact path: `src/mcp-stream-server.regression.test.ts`) with scenarios that would have caught the 2026-04-19 bug:
1. `initialize` + `tools/list` + `tools/call` × 3 against session-based transport — assert all three respond within 500ms each, no hang.
2. Same sequence against stateless transport — assert that subsequent requests after `initialize` (with or without `Mcp-Session-Id`) either succeed or return a spec-compliant error within 500ms. No indefinite hang.
3. `listChanged:false` capability assertion — parse `initialize` response, assert `capabilities.tools.listChanged === false`.
4. Concurrent-session test: two synthetic clients `initialize` in parallel, each runs 3 `tools/call`, assert no cross-contamination (client A's response never contains client B's `id`).
5. Session-TTL test: open session, wait past TTL, send a request with the old session ID, assert 404 session-not-found (not a 500).

**D-16. iOS verification is out-of-band**
Carsten tests the iPhone Claude-App manually after deploy. The harness cannot reproduce iOS-specific behaviors (the app ships closed-source). The DoD includes Carsten's signed-off manual verification: MCP connected + Claude.ai user messages work simultaneously for ≥ 3 minutes across ≥ 3 messages.

**D-17. Load verification — not in 4.5**
No synthetic load test for "500 concurrent sessions". Expected real load is ≤10. Measure in prod via the JSONL session log, revisit if spikes.

### Claude's Discretion

Details the planner/executor chooses within the locked decisions above:
- Exact zod schema shape for each of the 14 tools without metadata — derived from the tool's handler validation code (most tools already have `validateXxxArgs` functions in `src/mcp-tools/voice-*.ts`; port those zod schemas into `TOOL_META`).
- Error-code mapping between MCP JSON-RPC error codes and current handler error shapes (e.g. `UnknownToolError` → -32601 Method-not-found, `BadRequestError` → -32602 Invalid-params).
- Whether to share one `McpServer` instance across sessions with separate `Transport`s, or build both per-session. SDK behavior determines the answer; planner researches during phase.
- Session Map implementation: plain `Map` vs `Map` + `Set` for sweep efficiency. At expected scale (≤50), plain Map is fine.
- Log event names for new session lifecycle (e.g. `mcp_session_opened`, `mcp_session_closed`, `mcp_session_swept`). Follow existing `mcp_stream_*` naming convention.
- Test harness language: TypeScript (vitest, matches existing `src/*.test.ts` pattern).
- Whether to use the SDK's `Client` class in the bridge or hand-roll JSON-RPC over fetch. Default: SDK client unless it has blockers (e.g. Node fetch incompatibility).

## Specific Ideas

### Concrete target: `src/mcp-stream-server.ts` shape after 4.5

```ts
// Pseudocode — for planner orientation, not copy-paste target
interface Session {
  server: McpServer;
  transport: StreamableHTTPServerTransport;
  sessionId: string;
  createdAt: number;
  lastActivity: number;
}

const sessions = new Map<string, Session>();
const SESSION_TTL_MS = Number(process.env.MCP_STREAM_SESSION_TTL_MS ?? 30 * 60 * 1000);

// Sweep: every 60s, evict idle sessions, .close() their server+transport
setInterval(sweepIdle, 60_000).unref();

app.all('/', async (req, res) => {
  const sid = req.header('Mcp-Session-Id');
  let session: Session;
  if (sid && sessions.has(sid)) {
    session = sessions.get(sid)!;
    session.lastActivity = Date.now();
  } else {
    // Only initialize requests may create a new session
    if (!isInitializeRequest(req.body)) {
      res.status(400).json({ error: 'session_required' });
      return;
    }
    if (sessions.size >= 50) {
      res.status(503).json({ error: 'session_cap_reached' });
      return;
    }
    session = createSession();
    sessions.set(session.sessionId, session);
  }
  await session.transport.handleRequest(req, res, req.body);
});
```

### Concrete target: `voice-bridge/src/core-mcp-client.ts` shape after 4.5

```ts
// Pseudocode
import { Client } from '@modelcontextprotocol/sdk/client/index.js';
import { StreamableHTTPClientTransport } from '@modelcontextprotocol/sdk/client/streamableHttp.js';

class CoreMcpClient {
  private client: Client | null = null;
  async ensureConnected(): Promise<Client> {
    if (this.client) return this.client;
    const transport = new StreamableHTTPClientTransport(new URL(CORE_MCP_URL));
    this.client = new Client({ name: 'voice-bridge', version: '1.0.0' }, { capabilities: {} });
    await this.client.connect(transport);
    return this.client;
  }
  async callTool(name: string, args: unknown): Promise<unknown> {
    const c = await this.ensureConnected();
    const result = await c.callTool({ name, arguments: args as Record<string, unknown> });
    return result;
  }
  async close(): Promise<void> {
    if (this.client) await this.client.close();
    this.client = null;
  }
}

// One instance per live voice call — held in CallSession
```
