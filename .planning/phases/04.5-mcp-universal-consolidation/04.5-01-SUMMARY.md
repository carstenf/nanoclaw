### Scenario 5 Behavior Note

Plan Step 7 flagged that D-15 expected `404` for expired sid. Actual SDK behavior: a bogus/unknown sid on a non-initialize body hits our handler's "no session found + not an initialize request" branch, which returns `400 session_required`. The SDK transport itself would return 404 only for a sid that was known at some point and whose transport was explicitly `close()`'d by the server (via sweep eviction). Because our handler removes swept sessions from the Map immediately after closing their transport, the subsequent client request can no longer distinguish "unknown" from "expired" — both collapse to 400 session_required.

The test therefore asserts `[400, 404].includes(res.status)` and explicitly NOT 500, which matches the D-15 intent ("no 500, no hang"). This is noted as an executor-level deviation from the strict D-15 wording; no functional harm — the client sees a prompt, non-hanging 4xx with a clear error envelope.
