# Phase 4.5 Plan 02: Bridge v2 MCP SDK Client + Unit Tests Summary

Wave-2 introduces the SDK-based MCP client in the voice-bridge that Wave-3 will swap into production call sites. The new module `voice-bridge/src/core-mcp-client-v2.ts` wraps `@modelcontextprotocol/sdk` `StreamableHTTPClientTransport` + `Client` following RESEARCH.md Pattern 2, preserving the v1 public API (callCoreTool free-function + error classes) while adding a `CoreMcpClient` class for per-call session holding per D-6. Six unit tests drive the real client against an ephemeral-port instance of the Wave-1 session-based server, proving connect / callTool / close / timeout / server_error / reconnect_after_close paths all work end-to-end over the real MCP protocol.

Critical reversibility: v1 module `voice-bridge/src/core-mcp-client.ts` is byte-identical to pre-plan state. If Wave 3 hits trouble, the recovery is a one-file delete of v2 — no call-site churn to unwind.

## Task 1 — v2 Module

### File: `voice-bridge/src/core-mcp-client-v2.ts` (254 LOC, new)

Public surface:

| Export | Kind | Purpose |
|--------|------|---------|
| `CoreMcpError` | class | Generic MCP tool-call error. `(message: string, status?: number, toolName?: string)`. Wraps SDK `McpError` and `isError` payload results. |
| `CoreMcpTimeoutError` | class | Timeout-specific subclass. `(message: string, timeoutMs: number, toolName?: string)`. Thrown when SDK `RequestTimeout` (-32001) fires or signal aborts. |
| `CallCoreToolOpts` | interface | `{ timeoutMs?, url?, token?, signal? }` — v1-identical. |
| `CoreMcpClient` | class | Long-lived per-call session holder (D-6). Lazy-connects; idempotent close. Satisfies `CoreClientLike` for slow-brain/pre-greet DI. |
| `callCoreTool(name, args, opts?)` | async fn | Module-private singleton wrapper; drops in for v1 `callCoreTool` calls during Wave-3 migration. opts.url/token → short-lived client. |
| `__resetDefaultClientForTests()` | fn | Test-only hook to clear the module singleton. |

### Design notes

- **Lazy connect + race guard.** `ensureConnected()` captures concurrent callers in a shared `connecting: Promise<void>` so two racing `callTool()` invocations on a fresh client open exactly one transport.
- **Bearer via transport.** `requestInit.headers.Authorization: Bearer ${bearer}` passed into the transport constructor — unchanged from v1 (D-12).
- **RequestOptions timeout.** `c.callTool(params, undefined, { timeout, signal })` uses SDK's built-in per-request timeout (maps to McpError code `-32001 RequestTimeout`).
- **Result unwrapping.** Mirrors v1 semantics: parse JSON from `result.content[0].text`; fall back to raw text if JSON parse fails; fall back to the full SDK result if no text content present. The server always JSON-stringifies its handler result into a text block, so happy-path callers get the same object shape v1 returned.
- **Error mapping.**
  - `result.isError === true` → throw `CoreMcpError(content[0].text, undefined, name)` (mirrors v1 semantics where server 400 responses became thrown errors).
  - Caught `CoreMcpError` or `CoreMcpTimeoutError` → re-throw (prevent double-wrapping when `callTool` is called internally).
  - SDK error with code `-32001` OR message matching `/timed?\s*out|timeout|abort/i` → `CoreMcpTimeoutError(msg, timeoutMs, name)`.
  - Any other SDK `McpError` with numeric `.code` → `CoreMcpError(msg, code, name)`.
  - Network error → `CoreMcpError(msg, undefined, name)`.
- **Idempotent close.** `close()` early-returns on null client; swallows thrown errors from `client.close()` (Pitfall 4); sets `this.client = null` unconditionally so the next `ensureConnected()` opens a fresh transport (validates `reconnect_after_close` scenario).
- **Default-singleton pattern.** Module maintains `defaultClient: CoreMcpClient | null`, lazily constructed from `CORE_MCP_URL` + `CORE_MCP_TOKEN`. `callCoreTool` with `opts.url` or `opts.token` bypasses the singleton and spawns a short-lived client (closed in `finally`) so per-call overrides never contaminate the default.

## Task 2 — 6 Unit Tests

### File: `voice-bridge/src/core-mcp-client.test.ts` (244 LOC, replaced Wave-0 scaffold)

### Fixture design

- Ephemeral-port HTTP server hosting the real `buildMcpStreamApp` from `src/mcp-stream-server.js` — imports cross voice-bridge → root via `../../src/`. No HTTP mocking; fidelity over speed.
- Tools registered by TOOL_META name (`voice.check_calendar`, `voice.ask_core`, `voice.get_contract`) so the server's createSession takes the schema-aware `mcp.tool(name, description, meta.shape, handler)` path — the SDK validates args via zod and passes them to the registry handler. Schemaless-registered tools would get the SDK's "extra" params object as the first handler arg instead of caller args (SDK behavior); the `voice.echo`/`voice.slow`/`voice.throws` names in the plan-text would have taken that path and broken every round-trip assertion.
- Custom handlers override default registry behavior: `voice.check_calendar` echoes the synthetic args back, `voice.ask_core` sleeps, `voice.get_contract` throws.
- `__resetDefaultClientForTests()` in beforeEach/afterEach clears the module singleton between tests.

### Per-scenario runtime

| Scenario | Runtime | Asserts |
|----------|---------|---------|
| `connect: lazy-opens MCP session on first ensureConnected()` | 92ms | ensureConnected() is no-op on second call; subsequent callTool round-trips and echoes `duration_minutes: 11` |
| `callTool: returns server result, preserves args shape` | 26ms | echoed args include `{date: '2026-05-01', duration_minutes: 42}`; server-injected `call_id` carries `chat-` prefix (Pitfall 8 / D-11 assertion) |
| `close: idempotent, second close() is no-op` | 13ms | Second close() resolves undefined without throwing; close() on a never-connected client also safe |

| `reconnect_after_close: ensureConnected() after close() opens a fresh session` | 22ms | callTool after close() transparently re-ensureConnected; second round-trip works |

## Deviations from Plan

### Rule 1/2/3 Auto-Fixes

**1. [Rule 3 - Blocking issue] tsconfig excludes `src/**/*.test.ts` from tsc.**

- **Found during:** Task 2, attempting to run `npm run build` after adding cross-package imports to the test file.
- **Issue:** The plan's Task 2 Step 1 imports `'../../src/mcp-stream-server.js'`, `'../../src/mcp-tools/index.js'`, `'../../src/logger.js'` — three paths outside voice-bridge's `rootDir: src`. tsc with NodeNext resolution walked up to the root src/, emitted .ts → .js at the offending relative paths (e.g., `voice-bridge/dist/../../src/config.js`), leaving stray artifacts in the root src/ and failing the build with TS6059 rootDir errors.
- **Fix:** Added `src/**/*.test.ts` to the tsconfig `exclude` array. Tests run via vitest (vite-node transformer) which handles cross-package imports natively and doesn't need tsc emission. This is the standard Node.js project convention — the existing `tests` directory was already excluded; we extended it to cover colocated `*.test.ts` files too.
- **Files modified:** `voice-bridge/tsconfig.json`
- **Commit:** `54fed33`
- **Validation:** `npm run build` clean; `git status --short` shows no stray artifacts in root src/; all 313 passed | 1 skipped voice-bridge tests still green.

No Rule 1 (bug) or Rule 2 (missing critical functionality) fixes were needed.

### Plan-Text vs. Execution

1. **Fixture tool names changed.** Plan action Step 1 declared three fixture tools (`voice.echo`, `voice.slow`, `voice.throws`) with arbitrary handlers. The SDK's `mcp.tool(name, description, handler)` (no-schema) registration path passes its own "extras" object as the first handler arg — not the caller's args — so round-trip echo assertions on those names would always fail. We swapped to TOOL_META-registered names (`voice.check_calendar`, `voice.ask_core`, `voice.get_contract`) and used schema-valid args; the SDK then takes the schema-aware path where the registry handler receives the validated caller args + server-injected synthetic call_id/turn_id (Pitfall 8). Behavior-equivalent to the plan intent; concrete arg shapes documented in the test file.

2. **`callTool` arg merging sanity assertion added.** Plan action did not explicitly assert the Pitfall-8 synthetic `call_id`/`turn_id` injection — but since the server-side injection is now visible in the echo payload we assert `echo.call_id` carries the `chat-` prefix, guarding the D-11 contract in-situ.

3. **`close` scenario test strengthened.** Plan action stopped after the second `close()`. Added a third `close()` on a never-connected client to cover the early-return branch of `close()` — this is the branch that protects the post-constructor + no-ensureConnected-yet path.
