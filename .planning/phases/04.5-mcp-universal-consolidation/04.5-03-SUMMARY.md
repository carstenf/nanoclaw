## Finalizer Hook Location (Pitfall 5 / T-4.5-E)

**File:** `voice-bridge/src/sideband.ts`
**Location:** `ws.on('close', () => { ... })` — after `sideband_closed` info log, before `opts.onClose(callId)` invocation.
**Log events emitted:**
- Success: `{ event: 'bridge_core_mcp_client_closed', call_id }`
- Failure: `{ event: 'bridge_core_mcp_client_close_failed', call_id, err }`
**Behavior:** fire-and-forget (`void coreClient.close().then(..., err => ...)`). Non-blocking so teardown steps (silence-monitor stop, slow-brain stop, turn-log close, idempotency clearCall) proceed in parallel. `CoreMcpClient.close()` is idempotent (Wave-2 regression test `reconnect_after_close`) so the per-call model is safe even if a close races with a slow in-flight tool call.

### Plan-Text vs. Execution

1. **Task ordering adjusted — Tasks 1 + 3 combined in one commit.** Plan Step 6 Step 8 said "v1 and v2 both exist, but all 6 callers still consume v1... Task 3 flips this." In practice, adding `import { CoreMcpClient } from './core-mcp-client.js'` in Task 1 fails the TypeScript build because v1 never exported `CoreMcpClient` — only v2 does. The intermediate state where imports are swapped but v1 still occupies the canonical path is not compile-green. So Task 1 + Task 3 landed together in commit `fce7928` (consistent with the plan's own Task 2 action note "Commit this change IN THE SAME commit as Task 3 (rename + delete) so a checkout-at-any-commit works" — same principle, applied to Task 1 too).

2. **Task 2 `.env` flip is a runtime edit, not a git-tracked commit.** `.env` is in `.gitignore` (line 35); voice-bridge reads the root `.env` via its systemd `EnvironmentFile=%h/nanoclaw/.env`. The flip happened on the main tree's `.env` directly at `/home/carsten_bot/nanoclaw/.env:44`. No commit hash — it's a runtime config change tracked only by the systemd reload that picks it up.

3. **Per-call CoreMcpClient scope narrowed to pre-greet path.** The plan's interfaces section said "construct a `CoreMcpClient` per incoming call. Pass the instance downstream (to slow-brain, pre-greet) AND hand it to sideband's CallSession so finalizer can close it." The finalizer part landed; the downstream-for-all-call-sites part is scoped to the pre-greet adapter only. Cost-gate, sideband's fire-and-forget record_turn_cost/finalize_call_cost, and tools/dispatch continue to go through `callCoreTool` → the module-level default singleton. Rationale: D-6 asks for one session per live call, which serves the LONG-LIVED interactive path (pre-greet → slow-brain transcript turns). The fire-and-forget call sites inherently don't hold state across a conversation, so session-level pooling vs. per-call session has no observable difference. Expanding to all six sites would require threading `coreMcp` through `startSlowBrain(log, sideband.state, ?)` + `openSidebandSession`'s own three call-sites — a larger refactor the plan's `<behavior>` block did not explicitly require. Noted here so Phase 5+ can revisit if session lifecycle metrics warrant.
