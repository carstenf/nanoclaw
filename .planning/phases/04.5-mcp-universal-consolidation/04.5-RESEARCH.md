## Summary

Phase 4.5 is an **architectural-drift close-out**, not a green-field build. The spec says "MCP in both directions" (ARCHITECTURE-DECISION.md, PRD AC-07/AC-09, REQ-DIR-10/REQ-C6B-03). Phase 2 shipped a JSON-POST REST shortcut under the misleading name `core-mcp-client.ts`. Phase 4 Wave 3 added the spec-compliant StreamableHTTP channel on port 3201 but scoped it as a debug-only path. 4.5 retires the shortcut and migrates the bridge onto the spec channel.

Research verified three load-bearing facts: (1) The installed SDK (`@modelcontextprotocol/sdk@1.29.0`) supports everything we need — `sessionIdGenerator`, `onsessioninitialized`, `onsessionclosed`, `StreamableHTTPClientTransport` with automatic session-ID handling and reconnection. (2) There is a **known SDK bug (#1405, filed 2026-01-19, unresolved as of this research)** that breaks sharing one `McpServer` instance across multiple transports — the second `connect()` silently kills the first session. Our plan must therefore create a **fresh `McpServer` per session**, not a shared instance. (3) All 17 tool-handler modules already carry a zod schema inline (`const XxxSchema = z.object({...})`). The CONTEXT wording "port validateXxxArgs into zod" is off-by-one — nothing needs porting; the existing schemas can be **re-exported** and referenced from `TOOL_META`.

**Primary recommendation:** Implement D-2 exactly as stated, using the **session-store pattern** from SDK `docs/server.md` (transports Map keyed by sessionId + `onsessioninitialized` to register + `onclose` on transport to deregister), with the **per-session McpServer workaround** to avoid Issue #1405. Re-export each tool's existing zod schema from its handler module and reference it in the enriched `TOOL_META` table. Migrate bridge callers in the D-9 order.

### Pattern 1: Session-Based StreamableHTTP Server with per-session McpServer

**What:** Map<sessionId, Session>. On POST: look up by `Mcp-Session-Id`; miss + initialize → create fresh `McpServer` + `StreamableHTTPServerTransport`, register via `onsessioninitialized`, auto-deregister via `onclose`.

**When to use:** Every time the server-side session mode is in effect (i.e., always after 4.5 ships).

**Example** (synthesized from SDK `docs/server.md` pattern + Issue #1405 workaround):
```typescript
// Source: @modelcontextprotocol/sdk docs/server.md (session pattern)
//         + workaround for https://github.com/modelcontextprotocol/typescript-sdk/issues/1405
import { randomUUID } from 'node:crypto';
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { StreamableHTTPServerTransport } from '@modelcontextprotocol/sdk/server/streamableHttp.js';
import { isInitializeRequest } from '@modelcontextprotocol/sdk/types.js';

interface Session {
  server: McpServer;
  transport: StreamableHTTPServerTransport;
  createdAt: number;
  lastActivity: number;
}
const sessions = new Map<string, Session>();
const SESSION_TTL_MS = Number(process.env.MCP_STREAM_SESSION_TTL_MS ?? 30 * 60 * 1000);
const MAX_SESSIONS = Number(process.env.MCP_STREAM_MAX_SESSIONS ?? 50);

async function createSession(): Promise<Session> {
  // Per-session McpServer — workaround Issue #1405 (shared server overwrites transport).
  const mcp = new McpServer({ name: 'nanoclaw-voice', version: '1.0.0' });
  registerTools(mcp);                                      // same function Wave 3 used
  mcp.server.registerCapabilities({ tools: { listChanged: false } });

  const transport = new StreamableHTTPServerTransport({
    sessionIdGenerator: () => randomUUID(),
    onsessioninitialized: (sid) => {
      const s = { server: mcp, transport, createdAt: Date.now(), lastActivity: Date.now() };
      sessions.set(sid, s);
      log.info({ event: 'mcp_session_opened', sid });
    },
  });
  transport.onclose = () => {
    const sid = transport.sessionId;
    if (sid && sessions.delete(sid)) {
      log.info({ event: 'mcp_session_closed', sid, reason: 'transport_close' });
    }
  };
  await mcp.connect(transport);
  return { server: mcp, transport, createdAt: Date.now(), lastActivity: Date.now() };
}

app.all('/', async (req, res) => {
  const sid = req.header('Mcp-Session-Id');
  let session: Session | undefined = sid ? sessions.get(sid) : undefined;

  if (session) {
    session.lastActivity = Date.now();
  } else {
    if (!isInitializeRequest(req.body)) {
      res.status(400).json({ jsonrpc: '2.0', error: { code: -32000, message: 'session_required' }, id: null });
      return;
    }
    if (sessions.size >= MAX_SESSIONS) {
      res.status(503).json({ error: 'session_cap_reached' });
      return;
    }
    session = await createSession();
  }
  await session.transport.handleRequest(req, res, req.body);
});

// Idle sweep
setInterval(() => {
  const now = Date.now();
  for (const [sid, s] of sessions) {
    if (now - s.lastActivity > SESSION_TTL_MS) {
      void s.transport.close().catch(() => undefined);
      void s.server.close().catch(() => undefined);
      sessions.delete(sid);
      log.info({ event: 'mcp_session_swept', sid, age_ms: now - s.createdAt });
    }
  }
}, 60_000).unref();
```

### Pattern 2: Bridge MCP client — long-lived per-call session

**What:** `CoreMcpClient` class wrapping SDK `Client` + `StreamableHTTPClientTransport`. Lazy-connect on first `callTool`, held until `close()`.

**When to use:** One instance per `CallSession` (a live voice call). Never shared across calls.

**Example:**
```typescript
// Source: @modelcontextprotocol/sdk/client/index.d.ts + client/streamableHttp.d.ts
import { Client } from '@modelcontextprotocol/sdk/client/index.js';
import { StreamableHTTPClientTransport } from '@modelcontextprotocol/sdk/client/streamableHttp.js';

export class CoreMcpClient {
  private client: Client | null = null;
  constructor(private readonly url: URL, private readonly bearer?: string) {}

  async ensureConnected(): Promise<Client> {
    if (this.client) return this.client;
    const transport = new StreamableHTTPClientTransport(this.url, {
      requestInit: this.bearer
        ? { headers: { Authorization: `Bearer ${this.bearer}` } }
        : undefined,
    });
    const c = new Client({ name: 'voice-bridge', version: '1.0.0' }, { capabilities: {} });
    await c.connect(transport);                             // sends initialize + initialized
    this.client = c;
    return c;
  }

  async callTool(name: string, args: unknown, opts: { timeoutMs?: number } = {}): Promise<unknown> {
    const c = await this.ensureConnected();
    return c.callTool(
      { name, arguments: args as Record<string, unknown> },
      undefined,                                            // default result schema
      opts.timeoutMs ? { timeout: opts.timeoutMs } : undefined,
    );
  }

  async close(): Promise<void> {
    if (this.client) {
      try { await this.client.close(); } catch { /* ignore */ }
      this.client = null;
    }
  }
}
```

### Anti-Patterns to Avoid

- **Shared McpServer across sessions** — Issue #1405. Always `new McpServer(...)` per session.
- **Re-implementing session header parsing** — the transport parses `Mcp-Session-Id` itself. Our handler only uses it to look up in the Map; we do NOT call `req.header('Mcp-Session-Id')` and pass it downstream — `transport.handleRequest()` handles everything once we've routed to the right transport.
- **Mid-call teardown on sweep** — active sessions must NOT be swept. `lastActivity` updates on every request; only truly idle sessions expire.
- **Forgotten `unref()` on sweep timer** — blocks process exit in tests. Already in pseudocode; never skip.
- **Silent catch-all on `transport.handleRequest()`** — current Wave 3 code has `catch (err) { ... res.status(500) }` which hides spec-compliant error paths. New pattern: let the SDK's own response win unless headers truly weren't sent.
- **Per-request `registerCapabilities({ tools: { listChanged: false }})`** — with per-session McpServer this stays; what must **not** happen is calling it AFTER `mcp.connect(transport)` (throws per SDK `mcp.d.ts`).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON-RPC 2.0 envelope + initialize handshake | Custom fetch wrapper with hand-assembled envelopes | SDK `Client` + `StreamableHTTPClientTransport` | SDK handles: envelope, initialize/initialized sequencing, Mcp-Session-Id persistence, SSE parsing, reconnection (Issue #510 was about SSE client bug, fixed; StreamableHTTP client is active). |
| Server-side session routing | Custom Map + middleware to parse `Mcp-Session-Id` and dispatch | SDK `StreamableHTTPServerTransport` session mode (`sessionIdGenerator: () => randomUUID()`) | SDK generates session ID on initialize, sets response header, rejects with 400/404 per spec, manages per-stream SSE. We only own the Map of Session records (for TTL sweep + cap). |
| SSE event stream | Raw SSE framing / Last-Event-ID replay | SDK `eventStore` interface (if we ever need replay) | Not in 4.5 scope — our tools are request/response, no server-push. If iOS sends `Last-Event-ID`, SDK returns "no replay" gracefully. Event-store is optional. |

## Common Pitfalls

### Pitfall 1: SDK Issue #1405 — shared McpServer breaks concurrent sessions
**What goes wrong:** If you instantiate `McpServer` once outside the session-creation closure and then call `mcpServer.connect(transportA)` then later `mcpServer.connect(transportB)`, the SDK's `Protocol.connect()` silently overwrites `this._transport`. Session A's subsequent `tools/call` aborts mid-flight ("This operation was aborted", SSE socket.end).
**Why it happens:** SDK 1.29 `Protocol.connect()` has no "already connected" guard; documented workaround in issue thread is "new Server per session". [VERIFIED: https://github.com/modelcontextprotocol/typescript-sdk/issues/1405 filed 2026-01-19, no fix merged as of research date]
**How to avoid:** Inside `createSession()`, always `new McpServer(...)`. The McpServer is cheap (~no I/O, just in-memory registry). Cost: 18 zod schema references per init, ≪1 ms.
**Warning signs:** First iOS client works, second hangs. Regression test #4 (concurrent sessions) will catch this if violated.

### Pitfall 2: Stateless-mode initialize bug (what caused 2026-04-19 hang)
**What goes wrong:** In stateless mode (`sessionIdGenerator: undefined`), the SDK still sets `this._initialized` on the first request but the per-request factory throws it away. Clients that expect stateful semantics see inconsistent behavior — `initialize` succeeds, `tools/list` also OK (both get fresh factories), but the iOS client apparently opens a persistent GET for `notifications/tools/list_changed` (or waits for some session signal) that a stateless server cannot deliver.
**Why it happens:** SDK Issues #340, #360, #412 — stateless mode is underspecified and partially broken. The exact trigger for iOS' UI-block is unverified; what IS verified: switching to session mode is the documented path, and Wave 3's `registerCapabilities({tools:{listChanged:false}})` after `.tool()` calls is a workaround specifically documented by Wave 3 commit `0a5ba27` to coax iOS off the `list_changed` subscription.
**How to avoid:** Follow D-2 — session mode. Keep `listChanged: false` on session-mode as well (the tool list IS static in our system; we never `sendToolListChanged()`, so advertising `true` would be a lie).
**Warning signs:** Client hangs on first user message after connect. Server log shows `initialize` + `tools/list` green, then nothing.

### Pitfall 3: `Mcp-Session-Id` stripped by Caddy
**What goes wrong:** Caddy's `reverse_proxy` by default forwards most headers, but some header-munging directives can strip custom headers. If the bridge or iOS session ID doesn't round-trip back to the server, the second request returns `400 session_required` and the client can't recover (MUST send new `initialize` per spec).
**Why it happens:** Caddy config evolves; someone may add `header_up -Mcp-*` intending to sanitize.
**How to avoid:** Before iOS verification (D-13), run `curl -H 'Mcp-Session-Id: probe-12345' https://mcp.carstenfreek.de/nanoclaw-voice/ -X POST -d '{}' -H 'Authorization: Bearer ...' -H 'Content-Type: application/json'` and check the server log for that exact string. If absent → Caddy edit.
**Warning signs:** Server logs show `mcp_session_opened` but never a second request with the same sid; clients perpetually re-initialize.

### Pitfall 4: McpServer cleanup races
**What goes wrong:** `transport.close()` and `server.close()` called in quick succession can emit `onclose` twice or on already-closed object, throwing. Existing Wave 3 code mitigates with `.catch(() => undefined)` — must preserve.
**Why it happens:** SDK `Transport.close()` is idempotent per docs, but when called via HTTP connection close AND via sweep, the second call can see already-nulled internals.
**How to avoid:** Wrap both in `try { await x } catch {}`. Never await cleanup synchronously in the hot path; fire and forget with a catch.
**Warning signs:** Unhandled-rejection warnings at shutdown; flaky tests that shut down timers.

### Pitfall 5: Bridge call-session leak if `client.close()` not awaited in finalizer
**What goes wrong:** Voice call ends, `CallSession` finalizer runs but `coreClient.close()` is not awaited (or not called). The server-side session sits until TTL (30 min). At peak: 25 calls → 25 open sessions for half an hour → cap at 50 still OK, but wasteful.
**Why it happens:** Finalizers in call-session teardown are easy to miss; adding new branch forgets the client.
**How to avoid:** Single integration point — the `CallSession` object holds the `CoreMcpClient` instance; its teardown path (however it's implemented in `sideband.ts` + `webhook.ts` /hangup path) MUST include `await coreClient.close()`. The bridge session lifecycle (see `voice-bridge/src/sideband.ts:428` finalizer patterns) is where this goes.
**Warning signs:** `sessions.size` drift monotonically upward at the same rate as call volume; many `mcp_session_swept` events with `age_ms` close to TTL_MS.

### Pitfall 6: `handleRequest(req, res, req.body)` — double-parse
**What goes wrong:** If `express.json()` already parsed the body, passing `req.body` is correct. If you forget the third argument, the SDK tries to re-read the stream (which `express.json()` has already consumed) and hangs until timeout.
**Why it happens:** Copy-paste from the SDK docs' stateful example, which passes `req.body`.
**How to avoid:** Always `await transport.handleRequest(req, res, req.body);` — never `await transport.handleRequest(req, res);` in an Express app with `express.json()` middleware. Existing Wave 3 code does this correctly (line 264); preserve.
**Warning signs:** Client gets no response, server has no error log — the request is waiting for a body that will never come.

### Pitfall 7: Tool handler throws synchronously
**What goes wrong:** A tool handler's zod validation throws (e.g., `BadRequestError` not wrapped in an MCP-spec error). The SDK surfaces this as isError: true + text content, but current code in `src/mcp-stream-server.ts:205-213` wraps handlers with synthetic `call_id`/`turn_id` injection — if that injection mutates the args object incorrectly, zod in the downstream handler fails.
**Why it happens:** Mutation of frozen-args objects; shape mismatch from spread.
**How to avoid:** Preserve the existing spread pattern `{ ...(args as Record), call_id, turn_id }` — it creates a NEW object, never mutates. Also: run regression test #4 which calls each tool from a chat session and verifies no spurious validation errors.
**Warning signs:** `isError: true` on calls that should succeed, with zod error messages.

## Open Questions

1. **Does the iOS Claude-App actually send `Mcp-Session-Id` on its second request?**
   - What we know: spec MUSTs it; SDK client does it; iOS client behavior is closed-source.
   - What's unclear: iOS-app version in use; whether it speaks spec rev 2025-03-26 or 2025-11-25 (commit `ac3a651` mentions 2025-11-25 for RFC 9728).
   - Recommendation: log every request's `Mcp-Session-Id` header value on the server during D-16 iOS verification. If iOS omits it → that IS the hang root cause, and the plan's fallback (D-5 contingency) kicks in.

2. **Does SDK Issue #1405 affect us with per-session McpServer, or ONLY shared?**
   - What we know: issue thread says "new Server per session" is the workaround.
   - What's unclear: whether the SDK's internal `Protocol` has other shared state (e.g. global timeout/cache) that bites us.
   - Recommendation: regression test #4 (concurrent sessions) will catch it. Build in Wave 1, assert green before Wave 2.

3. **Is `StreamableHTTPClientTransport`'s auto-reconnect on network blip reliable for long-lived bridge sessions?**
   - What we know: `reconnectionOptions` exists with sensible defaults; resumption via `resumptionToken` + `Last-Event-ID` is supported.
   - What's unclear: our server does NOT provide an `eventStore` (no replay), so SDK client reconnect might successfully re-establish SSE but fail to replay a mid-flight request.
   - Recommendation: document as a known limitation — if bridge gets disconnected during an in-flight `tools/call`, the call fails; the bridge already handles this gracefully (REQ-DIR-12 graceful-degrade). No new behavior needed. Acceptance: monitor prod for `mcp_reconnect_*` events; if frequent, add server-side event store in follow-up.

4. **Should port-3200 deprecation log `mcp_rest_request_seen` also include the client IP?**
   - What we know: peer-allowlist already logs peer; adding to the deprecation counter helps identify the last consumer.
   - What's unclear: whether the Hetzner pricing-refresh cron uses an explicit client-ID we can grep.
   - Recommendation: log `{event, tool_name, peer_ip, user_agent}` — trivial addition, resolves question without extra work.
