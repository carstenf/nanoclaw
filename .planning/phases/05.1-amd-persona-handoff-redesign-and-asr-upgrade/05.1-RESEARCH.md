## 2. Defect #6 — Persona Handoff Root Cause (CRITICAL)

### 2.1 The smoking gun (trace evidence)

`task5-traces/v6-persona-swap-failed.jsonl`:

- **t=11672ms:** Model emits `amd_result={"verdict":"human"}` — classifier fires `onHuman` in `webhook.ts:279-303`
- **t≈11800ms** (approx, between trace entries): `updateInstructions(state, persona, log)` in `webhook.ts:296` sends `{type:'session.update', session:{instructions: <CASE2_OUTBOUND_PERSONA>}}` to the sideband WS
- **t=11879ms:** OpenAI returns:
  ```json
  {"type":"error","event_id":"event_DWpQrwwQuWRl13BDe71BA",
   "error":{"type":"invalid_request_error","code":"missing_required_parameter",
            "message":"Missing required parameter: 'session.type'.","param":"session.type","event_id":null}}
  ```
- **t=11879ms:** `function_call_output` for amd_result is posted (conversation continues with AMD persona still active)
- **t=14352ms:** `response.created` (scheduled by webhook.ts:298 via `setTimeout(requestResponse, GREET_TRIGGER_DELAY_OUTBOUND_MS=2500ms)` — 11879 + 2500 ≈ 14379, matches within timer drift)
- **t=14692ms onward:** Model speaks as **restaurant-assistant** helper-mode, because the session instructions are STILL `CASE2_AMD_CLASSIFIER_PROMPT` ("Du bist in Detektions-Modus") and the model, now asked to respond via `response.create`, improvises a role consistent with the conversation history (user said "hier Restaurant Bellavista" — model assumes it is the one being addressed).

[VERIFIED: v6-persona-swap-failed.jsonl line 27; confirmed by cross-reference with webhook.ts:294-299 and sideband.ts:605-630]

### 2.2 Root cause

The OpenAI Realtime API GA (General Availability, current 2026) requires `session.type` to be set on every `session.update` client event. This is because the **same** `session.update` client event is now used to configure both *speech-to-speech* sessions (`type: "realtime"`) and *transcription-only* sessions (`type: "transcription"`) — the type discriminator is mandatory. [CITED: https://developers.openai.com/api/docs/guides/realtime-conversations]

Our existing `updateInstructions()` in `sideband.ts:627-628`:

```typescript
state.ws.send(JSON.stringify({ type: 'session.update', session }))
```

where `session` is `{ ...extraSession, instructions }`. Missing: `type: 'realtime'` inside the `session` object.

**Why did this ever work before Phase 05.1?** The only prior use-site for `updateInstructions` at scale is:
1. `pre-greet.ts` for Case-6b inbound (Phase 3) — Slow-Brain pushes instructions when a pre-greet hint is produced
2. Cost hard-stop farewell (Phase 4, `sideband.ts:433`)

Both work because either (a) they run against a pre-GA Realtime endpoint, (b) OpenAI treats a missing `type` as a soft warning rather than a hard error in some regions/versions, or (c) the error was always there but went unobserved because nothing depended on the update actually landing — pre-greet instructions are *best-effort* (DIR-27 graceful degrade) and cost-farewell is followed immediately by ws.close (error fires, but farewell timer fires ws.close anyway).

Phase 05.1 is the first consumer that depends on `updateInstructions` having a **deterministic, post-update-observable effect** on the next `response.create`. The bug was always latent; only Case-2 exposes it.

[ASSUMED: precise timing of the GA rollout requiring session.type is not in public changelog; the error pattern matches numerous community reports since late 2025. Production impact: pre-greet instructions may have silently failed to land for existing Case-6b users too, but with cheap cost (model falls back to CASE6B_PERSONA floor). Planner should verify by grepping logs for `invalid_request_error.*session.type` in `/tmp/spike-a-trace-*.jsonl` or equivalent production JSONL.]

### 2.4 Fix — Layer 1 (Option 1, the primary)

**File:** `voice-bridge/src/sideband.ts`
**Function:** `updateInstructions(state, instructions, log, extraSession)` at line 605-640
**Change:** Ensure `type: 'realtime'` is in every outbound session.update payload.

Current line 619:
```typescript
const session: Record<string, unknown> = { ...extraSession, instructions }
```

Fixed:
```typescript
const session: Record<string, unknown> = { type: 'realtime', ...extraSession, instructions }
```

(Putting `type` first makes it explicit and callers passing `type` in extraSession can still override, though no production caller does.)

**Why this is correct:** `session.type='realtime'` tells OpenAI the session is a speech-to-speech session (the mode our `/accept` creates), not a transcription-only session. The API-reference example from the official docs shows this exact shape: `session: { type: "realtime", instructions: "Be extra nice today!" }` [CITED: community.openai.com session.type discussion].

**Test (vitest):**

```typescript
it('updateInstructions sends session.update with session.type="realtime" (defect #6 layer 1)', () => {
  const wsSent: string[] = []
  const state = { ready: true, ws: { send: (s: string) => wsSent.push(s) }, ... }
  updateInstructions(state, 'new instructions', log)
  const msg = JSON.parse(wsSent[0])
  expect(msg).toEqual({ type: 'session.update', session: { type: 'realtime', instructions: 'new instructions' } })
})
```

And a regression test that asserts **no `type: 'session.update'` payload is sent without `session.type: 'realtime'`** — run over all existing updateInstructions callers.

### 2.5 Fix — Layer 2 (Option 2, defense-in-depth)

Even after the session.update lands, hypothesis H1 (conversation-history contamination) is still possible — the conversation up to the verdict contains:
1. System: CASE2_AMD_CLASSIFIER_PROMPT (= "you are in detector mode")
2. User turn 1: (whatever the callee said)
3. Assistant: `amd_result={verdict:human}` function_call + function_call_output

The model, on the next `response.create`, sees: old system message + user opened with "hier Restaurant Bellavista" + the new instructions in the session floor. The LLM may still prefer the conversation cue over the floor instructions because conversational context is strong.

**Mitigation:** After `updateInstructions` returns true, immediately inject a synthetic `conversation.item.create` with role=user, content:

```
[System-Hinweis: AMD-Verdict war 'human'. Der Anruf läuft jetzt im Reservierungs-Modus.
Beginne bitte mit der Begrüßung gemäß deiner neuen Anweisungen.]
```

This synthetic user item gives the model an unambiguous prompt to "switch modes" that is aligned with the new instructions, breaking the conversational context that was implicitly established by the callee's opening greeting. [CITED: developers.openai.com/api/docs/guides/realtime-conversations — "system messages can be added at any point in the conversation… for smaller updates (e.g. 'the user is now asking about a different topic'), use system messages"]

**Note on role choice:** `role=system` in `conversation.item.create` is supported ([CITED: OpenAI community: "Message items of role system support only input_text content"]), but community reports some cases where assistant-role injections were ignored ([CITED: https://community.openai.com/t/bug-realtime-conversation-item-create-as-assistant-role-ignored-in-next-conversations/1103991]). Safer choice: `role=user` with a bracketed directive text, which the model reliably treats as a turn to respond to.

Implementation sketch (in `webhook.ts` onHuman closure):

```typescript
if (ctxRef) {
  updateInstructions(ctxRef.sideband.state, persona, log)
  // Defense-in-depth: synthetic user directive breaks conversational context
  // inherited from AMD detector mode (pre-verdict callee utterance).
  ctxRef.sideband.state.ws?.send(JSON.stringify({
    type: 'conversation.item.create',
    item: {
      type: 'message',
      role: 'user',
      content: [{ type: 'input_text', text: '[System-Hinweis: AMD-Verdict war human. Beginne jetzt mit der Begrüßung gemäß deiner neuen Anweisungen.]' }],
    },
  }))
  setTimeout(() => {
    if (ctxRef) requestResponse(ctxRef.sideband.state, log)
  }, GREET_TRIGGER_DELAY_OUTBOUND_MS)
}
```

**Why NOT a full hard reset (Option 1 from DEFECTS.md):** a fresh `realtime.calls.accept` re-negotiates SIP, re-negotiates audio codec, introduces 500-1500ms of audio dropout (verified live: Phase 2 SIP accept typically 300-700ms to sideband ready, plus new RTP bind-and-sync). This is the wrong tradeoff: we are trying to *reduce* latency between AMD-verdict and first caller utterance, not add it.

### 2.6 Testing approach for defect #6

**Note on Realtime audio format:** OpenAI Realtime internally works at **24kHz** PCM (or 24kHz opus), NOT 8kHz. The 8kHz telephony audio from Sipgate is upsampled before it reaches OpenAI's Realtime API. [VERIFIED: SESSION_CONFIG.audio.output.format in our code at `config.ts:295` confirms 24kHz]. So the "at 8kHz telephony bandwidth" question is actually "how well do these models handle audio that was originally 8kHz and upsampled to 24kHz for the API" — upsampling doesn't add information, so the effective bandwidth is still telephony-limited. GPT-4o-transcribe's improved acoustic model is more robust to this low-information audio than whisper-1. [ASSUMED: no public OpenAI benchmark specifically on 8kHz-upsampled-to-24kHz German audio; inference from documented FLEURS improvements and general GPT-4o audio quality.]

### 3.3 Recommendation: gpt-4o-mini-transcribe

Choose `gpt-4o-mini-transcribe` over `gpt-4o-transcribe` because:

1. **Latency is critical here.** The classifier + transcript-cue regex in `amd-classifier.ts` operate on the first ≤6s post-accept. Lower transcription latency gives the regex-cue gate more chances to fire before the 30s silence timer. Full `gpt-4o-transcribe` adds latency without clear benefit for short restaurant-greeting utterances.
2. **Cost is not the driver,** but mini is cheaper and aligns with the Phase 4 cost caps (CAP_PER_CALL_EUR in `cost/gate.ts`).
3. **Accuracy improvement over whisper-1 is substantial** — the FLEURS WER improvement documented by OpenAI covers the dominant pathology (word-level hallucination on short utterances).

### 3.6 VAD/turn-detection interaction (research question #6)

Will the ASR upgrade affect the VAD silence timer assumption (CASE2_VAD_SILENCE_MS=30000ms)?

**Short answer: No, not materially.** VAD and transcription are **independent** pipelines in OpenAI Realtime.

- `server_vad` runs on the input audio buffer directly (uses its own VAD model, not the transcriber). It emits `input_audio_buffer.speech_started` and `speech_stopped` events based on audio signal.
- Transcription (`whisper-1` or `gpt-4o-mini-transcribe` or `gpt-4o-transcribe`) runs in parallel on the same audio buffer, emitting `conversation.item.input_audio_transcription.completed` (and with the newer models, also `.delta`).

Our VAD timers fire on the `speech_started`/`speech_stopped` events [VERIFIED: `amd-classifier.ts:179-211`], NOT on transcription-completed events. The ASR model change does not affect VAD timing.

`gpt-4o-mini-transcribe` may deliver `completed` events slightly FASTER than whisper-1 (because it can stream deltas), but our code does not use this — we wait for `completed`. No change needed to CASE2_VAD_SILENCE_MS.

[VERIFIED: independence of VAD and transcription via OpenAI Realtime docs; VAD config shape `session.audio.input.turn_detection` is separate from `session.audio.input.transcription` — both are per-call configs on the same audio stream.]

Mapping (per DEFECTS.md §4 recommendation, cross-verified with semantic intent):

| AMD reason | → zod `prev_outcome` | Rationale |
|-----------|---------------------|-----------|
| `amd_result` | `voicemail` | Model said verdict=voicemail (or unknown → safe default) |
| `cadence_cue` | `voicemail` | Long uninterrupted speech = mailbox greeting |

| `transcript_cue` | `voicemail` | Regex matched mailbox phrase |

**Alternative consideration:** Should `silence_mailbox` map to `no_answer` instead? 1. From the retry-ladder perspective, silent-pickup and normal voicemail are the same (retry in 5min)
2. The `no_answer` case is NOT an AMD classifier outcome — it's a Sipgate REST error path (`originate_failed` with cause=`no-answer`), handled upstream in `outbound-router.ts`, not in `amd-classifier.ts`
3. So there's no ambiguity: all four AMD reasons → `voicemail`; `no_answer` fires from a completely different code path

### 5.1 Current state (the bug)

`src/mcp-tools/voice-start-case-2-call.ts:202-217`:

```typescript

     (target_phone, calendar_date, attempt_no, scheduled_for, idempotency_key,
      originating_call_id, restaurant_name, created_at)
```

PK is `(target_phone, calendar_date, attempt_no)` per schema ([VERIFIED] DEFECTS.md §5 quotes the UNIQUE constraint error). Two distinct idempotency_keys for the same (phone, date) both try to insert attempt_no=1 → UNIQUE constraint fails on the second.

**Why is this an issue?** Carsten makes 2 reservations at the same restaurant on the same day — e.g., lunch at 13:00 for 2, dinner at 19:30 for 4. These have DIFFERENT idempotency_keys (the key includes `requested_time` and `party_size`). They are legitimate distinct bookings, not duplicates. But with hardcoded attempt_no=1, the second `voice_start_case_2_call` invocation rejects with SqliteError.

### 5.2 Recommended fix — better-sqlite3 transactional read-then-insert

[VERIFIED via project grep: existing codebase uses `db.transaction(() => ...)()` pattern; example: `src/mcp-tools/voice-case-2-retry.ts:155-184` uses the INSERT-OR-FAIL retry loop pattern already]

Research question #4 asked: `BEGIN; SELECT MAX+1; INSERT; COMMIT` vs `INSERT ... SELECT` with subquery?

**Recommendation: use `db.transaction()` wrapper with separate SELECT + INSERT statements.** This is the project idiom ([VERIFIED] `voice-case-2-retry.ts` already uses it). Embedding the MAX subquery directly in the INSERT VALUES clause is technically possible in SQLite but:
- Less readable
- Harder to debug (can't log the computed attempt_no for audit)
- Harder to combine with the existing duplicate-check flow (lines 166-197 do a SELECT for idempotency_key first)

**Note on race:** `voice-case-2-retry.ts` already handles PK collisions with a retry loop (lines 155-184). For `voice-start-case-2-call.ts`, a single-user Carsten setup has no real race (Carsten serializes his own bookings via Andy in `src/index.ts`). But defense-in-depth says: same pattern applies.

**Fix sketch (replace lines 199-221 of voice-start-case-2-call.ts):**

```typescript
const created_at = new Date(nowFn()).toISOString();
const scheduled_for = created_at;

let attempt_no = 1;
try {
  const tx = db.transaction(() => {
    const row = db.prepare(
      `SELECT COALESCE(MAX(attempt_no), 0) + 1 AS next_attempt_no
       FROM voice_case_2_attempts
       WHERE target_phone=? AND calendar_date=?`
    ).get(restaurant_phone, requested_date) as { next_attempt_no: number };
    attempt_no = row.next_attempt_no;

    db.prepare(
      `INSERT INTO voice_case_2_attempts
         (target_phone, calendar_date, attempt_no, scheduled_for, idempotency_key,
          originating_call_id, restaurant_name, created_at)
       VALUES (?, ?, ?, ?, ?, ?, ?, ?)`
    ).run(
      restaurant_phone,
      requested_date,
      attempt_no,
      scheduled_for,
      idempotency_key,
      call_id ?? null,
      restaurant_name,
      created_at,
    );
  });
  tx();
} catch (err) {
  logger.warn({ event: 'voice_start_case_2_db_insert_error', err });
  return { ok: false, error: 'internal' };
}
```

`better-sqlite3`'s `db.transaction()` wraps in BEGIN/COMMIT automatically and is synchronous (no promise dance). [VERIFIED: better-sqlite3 docs + existing project usage pattern]

So even with the fix, a same-day flood of >5 initial-call attempts will trip the same cap — good. The cap applies to total attempts per (phone, date), retries + initial calls combined. ## 8. Common Pitfalls for Phase 05.1

### Pitfall 1 — Fixing only H3 (session.type) without H1 defense-in-depth

**What goes wrong:** session.update now lands, but model still mis-reads the conversational context (callee's opening "hier Restaurant Bellavista"), producing restaurant-assistant behavior despite updated instructions.

**Why it happens:** LLMs are sensitive to conversational priors. With the old CASE2_AMD_CLASSIFIER_PROMPT fading and the new CASE2_OUTBOUND_PERSONA arriving, the model may still weight the recent user turn more than the new system instructions.

**How to avoid:** implement Layer 2 (synthetic `conversation.item.create role=user`) alongside Layer 1. Both fixes ship together, tested together.

**Warning signs:** in live-verification, bot's first word is "Hallo!" (helper greeting) instead of "Guten Tag, hier NanoClaw im Auftrag von Carsten Freek" (caller greeting).

### Pitfall 2 — Observability blind spot on session.update rejection

**What goes wrong:** if a future session.update ever drops a required parameter again, the failure is silent (error logged by OpenAI, but our log just shows the outbound send succeeded).

**Why it happens:** our WS message-handler in `sideband.ts:294-537` ignores `error` events (no explicit case; falls through to "silent ignore" at line 529).

**How to avoid:** add explicit handler for `parsed.type === 'error'`. Log at `log.error` with the error.code, error.message, and error.param. Ensure test 3 of §2.6 exists.

**Warning signs:** none visible to user; only caught by CI regression test or live-call log audit.

### Pitfall 3 — gpt-4o-mini-transcribe cost surprise

**What goes wrong:** Task 6 live calls burn more €/min on transcription than Phase 4 cost caps anticipated.

**Why it happens:** gpt-4o-mini-transcribe pricing differs from whisper-1 (both are billed per minute of audio, but the GPT-4o family has higher per-minute rates).

**How to avoid:** [ASSUMED] before Task 6, check `src/cost-ledger.ts` and Phase 4 `CAP_PER_CALL_EUR` — confirm the cap still holds with the new pricing. If not, bump cap OR stay on whisper-1. Plan contingency: if cost cap trips on the first Task 6 call, fall back to `gpt-4o-transcribe`-less-streaming mode (OR add it as `/tmp` fallback envelope).

**Warning signs:** Phase 4 cost-accumulator fires `cost_soft_warn` earlier than expected.

### Pitfall 4 — Transaction-wrapper deadlock in same-file calls

**What goes wrong:** `db.transaction(() => ...)()` called inside an already-transactional context (e.g., another MCP tool's transaction) may deadlock or nest incorrectly.

**Why it happens:** better-sqlite3 transactions are reentrant but each call to `tx()` opens its own SAVEPOINT. If `voice-start-case-2-call` is ever called from within another transaction (currently it isn't), the SAVEPOINT management has to be sorted.

**How to avoid:** `voice-start-case-2-call` is called from a fresh MCP request handler (no enclosing transaction) — safe. Add code comment noting the assumption.

**Warning signs:** SqliteError "cannot start a transaction within a transaction" in tests or prod.

### Pitfall 5 — Synthetic user-item injection interacts with server_vad.create_response

**What goes wrong:** our `conversation.item.create` synthetic user injection may not *itself* trigger a `response.create` (user input items via the client-event path don't go through VAD). We still need the explicit `requestResponse` call.

**Why it happens:** `server_vad.create_response: true` creates responses from VAD-detected speech_stopped events; injected items are not VAD-scoped.

**How to avoid:** after `conversation.item.create`, still call `requestResponse` with `GREET_TRIGGER_DELAY_OUTBOUND_MS=2500` delay (existing code pattern). Test 2 of §2.6 asserts both messages are sent in order.

**Warning signs:** session.update + item.create lands successfully but bot stays silent — nobody triggered a response.

### Pitfall 6 — Double amd_result handler invocation on session.update error

**What goes wrong:** if OpenAI returns `invalid_request_error` on session.update, our error is returned on the WS stream. If by coincidence, the classifier `.onHuman` is re-entrant (it isn't — `settleHuman` is guarded by `verdict !== 'pending'`), a retry could double-fire the persona push.

**Why it happens:** pure defense-in-depth consideration.

**How to avoid:** classifier is already idempotent [VERIFIED: `amd-classifier.ts:154-176` — `if (verdict !== 'pending' || stopped) return`]. Confirm test coverage exists in `amd-classifier.test.ts` for "amd_result fired twice → onHuman called once."

**Warning signs:** session.update sent twice in a row in the trace.

## 9. Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session.update retry-on-transient-error | Custom retry/backoff | Leave as-is; single-shot send | OpenAI's WS error is deterministic (schema issue, not transient). Retry on persistent schema error = infinite loop. |
| Conversation-history slicing / item.delete loop | Custom WS message tracker | Layer 2 synthetic user item + trust model to re-read instructions | community.openai.com/t/reset-conversation-in-realtime-api confirmed: manual delete-all loop is an anti-pattern; synthetic items + updated instructions is the documented way |
| ASR confidence filter (reject low-confidence transcripts) | Custom text-quality heuristics | Trust gpt-4o-mini-transcribe's output; regex on confirmed text | The AMD classifier already uses the `completed` event only; false-positive rate of the regex is controlled by the regex itself (see Spike-C corpus validation in Wave 0) |

## 13. Open Questions (RESOLVED)

### OQ-1. Is `session.type` required by our *specific* OpenAI endpoint (GA vs beta)?

**What we know:** the error in the trace is definitive for our endpoint (we see `missing_required_parameter: session.type`).
**What's unclear:** is this a recent rollout? Does our `OpenAI-Beta: realtime=v1` header in `sideband.ts:192-193` give us a grace period?
**Recommendation:** Add `type: 'realtime'` regardless — it's a GA-compliant shape and works on both GA and beta endpoints. No downside.

### OQ-2. What exactly is the "correct" synthetic-item role — user vs system?

**What we know:** community reports recommend `user` role for reliable effect; `role=system` is supported but "only for smaller updates."
**What's unclear:** in this context, is AMD-to-persona handoff a "small update" (system) or a "major change" (reserve for `session.update`)? Both session.update AND synthetic item is the belt-and-suspenders; the question is what text to put in the item.
**Recommendation:** Use `role=user` with a bracketed system-directive text ("[System-Hinweis: AMD-Verdict…]"). This leverages role=user's reliable-trigger property while semantically carrying a system directive.

### OQ-3. Should we add a `prompt` field to the transcription config for German restaurant vocabulary?

**What we know:** `prompt` is optional for gpt-4o-mini-transcribe/transcribe; helps for domain vocabularies.
**What's unclear:** is general restaurant German enough for whisper-1 hallucinations to go away, or do we need specific priming (e.g., "Die folgende Audioaufnahme ist ein deutsches Telefonat über Restaurant-Reservierung")?
**Recommendation:** Start without `prompt`. If Task 6 live calls show ASR still bad, add a prompt as a fallback. Plan documents this contingency.

### OQ-4. How to regression-test that pre-greet.ts (Case-6b) still works after the session.type fix?

**What we know:** pre-greet.ts uses `updateInstructions` on the Case-6b path. The fix applies uniformly (all session.updates get type:'realtime'), which SHOULD be backward-compatible.
**What's unclear:** did pre-greet.ts ever actually work correctly pre-fix, or was it silently failing too? Worth auditing prod JSONL for `event='error'` with `param: 'session.type'` in recent Case-6b calls.
**Recommendation:** Planner adds a task: grep `/tmp/spike-a-trace-*.jsonl` and `voice-bridge/logs` for error events with param=session.type BEFORE the fix lands, to establish a baseline. If pre-greet was also silently broken, that's additional evidence this fix is strictly an improvement, not a regression risk.
