# Phase 05.1 Plan 01: Defect #6 L1+L2 — post-AMD persona handoff (session.type + synthetic directive) Summary

One-liner: Fix OpenAI Realtime GA 2026 `session.update` rejection by injecting `session.type='realtime'` at the sideband layer and add a synthetic `conversation.item.create role=user` directive after AMD verdict=human to break conversational context contamination.

## What was built

Two-layer fix for Defect #6 (post-AMD persona handoff silent failure observed in Plan 05-03 Task 5 live calls):

**Layer 1 — root-cause spec conformance (`voice-bridge/src/sideband.ts:619`)**
The `updateInstructions()` function now prepends `type: 'realtime'` to every `session.update` payload. OpenAI Realtime GA 2026 requires this discriminator; without it the server returns `invalid_request_error(param='session.type')` and the CASE2_OUTBOUND_PERSONA swap silently fails, leaving the model running on CASE2_AMD_CLASSIFIER_PROMPT. Placed FIRST in the object literal so `extraSession` spread can still explicitly override (no production caller currently does).

**Layer 1 — observability (`voice-bridge/src/sideband.ts` WS onmessage)**
Added explicit `parsed.type === 'error'` handler that logs `event='session_update_rejected'` at ERROR level with `code`, `message`, `param`, `error_type`, and `openai_event_id`. Closes the Pitfall 2 silent-swallow gap that let defect #6 remain invisible in logs during live calls.

**Layer 2 — defense-in-depth (`voice-bridge/src/webhook.ts` onHuman closure)**
Between `updateInstructions(persona)` and `setTimeout → requestResponse`, inject a synthetic `conversation.item.create` with `role=user` and a bracketed directive text:

```
[System-Hinweis: AMD-Verdict war human. Der Anruf laeuft jetzt im Reservierungs-Modus. Beginne bitte mit der Begruessung gemaess deiner neuen Anweisungen.]
```

This breaks the conversational context inherited from `CASE2_AMD_CLASSIFIER_PROMPT` so the model cannot mis-read the callee's opening greeting ("Restaurant Bellavista") as evidence it should keep playing helper-mode. Text uses ASCII umlauts per Phase 2 CASE6B_PERSONA convention. The `conversation.item.create` does NOT itself trigger a `response.create` (Pitfall 5: VAD only scopes audio-derived items), so the explicit setTimeout→requestResponse remains unchanged.

Also added `dispatch.getAmdClassifier()` as a test-only accessor so the accept.test.ts can drive `classifier.onAmdResult('human')` end-to-end through the full `/accept` code path.
