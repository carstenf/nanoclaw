# Phase 05.1 Plan 03: Defect #4 — onVoicemail zod-valid retry args Summary

One-liner: webhook.ts Case-2 onVoicemail now constructs voice_case_2_schedule_retry args matching the Core zod schema (call_id + calendar_date + idempotency_key + prev_outcome='voicemail'), with a fail-fast guard that skips the retry and logs case_2_schedule_retry_missing_fields when required fields are missing from casePayload rather than sending zod-invalid empty-string fallbacks.

## What changed

### voice-bridge/src/webhook.ts

**New exported factory** `buildCase2OnVoicemailHandler` (pre-existing `registerAcceptRoute` signature unchanged):

- Accepts `callId`, `activeOutbound` (task_id + target_phone), `casePayload`, `coreMcpForAmd`, `openai`, `log`, `setAmdClassifier`
- Returns an async `(reason: AmdVoicemailReason) => Promise<void>` handler
- Maps all four AMD reasons through `amdReasonToPrevOutcome` to zod enum `'voicemail'`
- Fail-fast guard: if `casePayload.requested_date` or `casePayload.idempotency_key` is missing, empty, or non-string, logs `case_2_schedule_retry_missing_fields` with `has_calendar_date` + `has_idempotency_key` booleans and skips the `voice_case_2_schedule_retry` tool call
- On success path, sends zod-valid args: `{ call_id, target_phone, calendar_date, prev_outcome: 'voicemail', idempotency_key }`
- `voice_notify_user` call block is unchanged (preserved per RESEARCH §4.4 explicit)

**Wire change in /accept route:** the Case-2 `createAmdClassifier({ onVoicemail: ... })` config now uses `buildCase2OnVoicemailHandler({...})` instead of an inline async closure. All captured context is passed explicitly as factory params.

### voice-bridge/tests/webhook-case-2-voicemail.test.ts

New file; 13 passing tests:

- **Test 1 (×4)** — one per AMD reason (`amd_result` | `cadence_cue` | `silence_mailbox` | `transcript_cue`): asserts the handler calls `voice_case_2_schedule_retry` with `objectContaining({ call_id, target_phone, calendar_date, prev_outcome: 'voicemail', idempotency_key: /^[0-9a-f]{64}$/ })`; asserts the old buggy keys (`task_id`, `case_payload`) are absent; asserts `prev_outcome` is not the raw AMD reason.
- **Test 2 (×4)** — one per AMD reason: runs the captured args through `VoiceCase2ScheduleRetrySchema.safeParse()` (replicated schema kept in sync with `src/mcp-tools/voice-case-2-retry.ts:36-44`); asserts `success=true` for each of the 4 reasons. This closes the Bridge↔Core contract loop at build time, not runtime.
- **Test 3** — voice_notify_user regression guard: retry + notify both fire.
- **Test 4** — error resilience: `voice_case_2_schedule_retry` rejection is caught + logged as `case_2_schedule_retry_failed`; notify still fires.
- **Test 5a / 5b / 5c** — fail-fast: missing idempotency_key, missing requested_date, and empty-string idempotency_key all produce `case_2_schedule_retry_missing_fields` warn-log and NO retry tool call.

### voice-bridge/package.json + package-lock.json

Added `zod@^4` to `devDependencies` (test-only, no runtime dep). Justification: Test 2 needs the zod schema to validate handler output against the Core server contract; replicating the schema inline (per Plan instructions, to avoid pulling better-sqlite3 into the bridge test graph) requires zod at test time.

## Deferred Issues

**Same zod-shape bug exists at voice-bridge/src/outbound-router.ts:245-252.** The error-driven retry path in `reportBackCase2` still passes `task_id + case_payload + prev_outcome=verdict` — though it additionally passes `idempotency_key` + `calendar_date`, so Core would reject with `-32602` on `task_id` (extra field, non-strict object by default may accept) but crucially `prev_outcome` here is `voicemail_detected | line_busy | no_answer`, of which only `no_answer` matches the zod enum. The other two values ('voicemail_detected', 'line_busy') are rejected by `z.enum(['no_answer', 'busy', 'voicemail', 'out_of_tolerance'])`.

Out of scope for this plan (plan strictly targets webhook.ts onVoicemail per `<objective>`). Existing `outbound-router.test.ts` tests 4-1..4-3 assert the current (buggy) shape and would need updating in a follow-up plan. Flagged for 05.1-05 live verification to check whether any outbound-router error path fires during Scenario VOICEMAIL (likely not — AMD classifier is the active sink for voicemail outcomes in Case-2) or a dedicated follow-up (05.1-0X or Phase 05.2).

Recommended action: log this as a new item in Phase 05.1 blockers or as a new decimal plan 05.1-03b if the live-verify reveals the outbound-router path is ever exercised.
