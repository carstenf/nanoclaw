## Decisions Made

- **`db.transaction()` wrapper over INSERT-with-subquery:** per RESEARCH §5.2 — readable, debuggable (attempt_no variable available for logging), matches project idiom (voice-case-2-retry.ts).
- **No retry-on-collision loop (unlike voice-case-2-retry.ts):** that file needs the `INSERT OR FAIL` + `attempt_no++` loop because retries may race between concurrent schedulers. `voice-start-case-2-call` is only called via Andy in `src/index.ts` — serialized — so transactional SELECT+INSERT is sufficient and simpler. Matches RESEARCH §5.2 exactly.
- **Accept gaps in attempt_no:** PK enforces uniqueness, ordinal is audit-only (RESEARCH §9 "Don't Hand-Roll" explicitly rejects gap-filling). Test 5 (pre-seeded 1, 2, 5 → new=6) encodes this semantic as a test invariant.
- **Pitfall 4 code-comment invariant:** documented inline above the new `db.transaction()` block — `voice-start-case-2-call` only runs in fresh MCP request handlers with no enclosing transaction. Future callers from nested transaction contexts need to review to avoid `SqliteError: cannot start a transaction within a transaction`.
