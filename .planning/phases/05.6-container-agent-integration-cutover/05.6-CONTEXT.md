## Implementation Decisions

### Phase-06-specific decisions

- **D-26 — Real `defaultInvokeAgent` uses NanoClaw IPC pipeline.** Container-spawn happens through the existing `src/container-runner.ts` path used by other channels (WhatsApp, Discord). No new container-spawning machinery. Implementation may piggyback on the existing `andy-agent-runner` / per-group container reuse, treating the `call_id` as a synthetic group-key OR using the original triggering group's container if known. Planner decides exact mapping.
- **D-27 — Skill loading happens inside the container-agent, not in the MCP-tool handler.** The handler (`voice-triggers-init.ts`) calls into the agent with a structured prompt that says "load the `voice-personas` skill, render persona for case_type=X / direction=Y / counterpart=Z, return only the rendered string." Substitution + Du/Sie derivation happen inside the agent + skill, not in the MCP-tool body. This keeps the MCP-tool a thin shim; reasoning lives in the agent.
- **D-28 — Cutover sequence has explicit checkpoints between each step.** The phase plan must be `autonomous: false` (or contain explicit checkpoint tasks). Carsten gates: (a) start cutover (after 05.5 merged + verified), (b) synthetic test PASS gate, (c) inbound PASS gate, (d) outbound PASS gate, (e) flip-default gate, (f) cleanup-commit-now gate. Each gate is a manual decision; the phase does NOT proceed past a gate without explicit OK.
- **D-29 — Live test acceptance criteria are STRICT** (per plan-checker WARNING #4 fix): live tests must demonstrate at least one **case-specific behaviour** to PASS, not just "any non-error response". Inbound test must show Case-6b functionality (Kalender, Fahrzeit, or Memory-Lookup interaction). Outbound test must show Case-2 booking-attempt dialog (counterpart-question handling, hold-music tolerance, or readback). Generic "Hallo" + "Auf Wiederhören" alone is NOT a PASS.
- **D-30 — Cleanup commit is single-commit-atomic.** All deletes + flag removal + ENV removal in one commit. Reason: easier rollback (single `git revert`), easier review, clearer history. The planner must NOT split cleanup across multiple plans.
- **D-31 — Post-cleanup smoke is required.** After the cleanup commit, run a final synthetic 2-turn smoke test to confirm the single-path code (no flag, no slow-brain) still works. If smoke fails, the cleanup commit gets reverted; investigate root-cause; re-attempt.

### `defaultInvokeAgent` real implementation — sketch

```ts
// src/mcp-tools/voice-triggers-init.ts
async function realInvokeAgent({ call_id, case_type, call_direction, counterpart_label }, deps) {
  // 1. Determine container key (D-26): per-call synthetic group OR triggering-group's container
  const groupKey = mapCallToGroup(call_id, case_type, call_direction);

  // 2. Build agent prompt that instructs skill loading + rendering
  const agentPrompt = buildPersonaRenderPrompt({ case_type, call_direction, counterpart_label });
  // Prompt body: "Use the voice-personas skill. Load baseline.md and overlays/${case_type}-*.md.
  //               Derive anrede_form per the SKILL.md Du/Sie rule. Substitute placeholders.
  //               Return ONLY the rendered string as your final output, no explanation."

  // 3. Invoke through containerRunner (D-26)
  const result = await deps.containerRunner.runOnce(groupKey, agentPrompt, {
    timeout_ms: 4500,  // leaves 500ms headroom for Bridge's 5000ms MCP timeout (REQ-DIR-20)
  });

  // 4. Extract rendered string from agent output
  return { instructions: extractFinalString(result) };
}
```

The exact shape of `containerRunner.runOnce` may not exist today — planner should propose it as a new API on container-runner, or wrap the existing one-shot path used by Andy.
