# Phase 05.5 Plan 05: Cost-ledger trigger_type + per-trigger recording — Summary

**One-liner:** Adds a `trigger_type` column to `voice_turn_costs` (with `'turn'` default for backward-compat) and wires both new MCP-tools (`voice_triggers_init`, `voice_triggers_transcript`) to record one synthetic cost-ledger row per container-agent invocation, satisfying REQ-COST-06 without disrupting Phase-4 SUM aggregations.

## What Was Done

### Task 1 — Schema migration + RecordTurnCostSchema extension

- **`src/db.ts`** — Added a guarded `ALTER TABLE voice_turn_costs ADD COLUMN trigger_type TEXT NOT NULL DEFAULT 'turn'` block in `createSchema()`. Idempotent via `PRAGMA table_info()` check; safe on production DBs that already have the table from Phase 4.
- **`src/cost-ledger.ts`** — Schema mirror in `createSchema()` (used by in-memory tests) gets the column inline in the CREATE TABLE statement. The `VoiceTurnCostRow` interface gains an optional `trigger_type?` field. `insertTurnCost()` populates the column with a `'turn'` default when `row.trigger_type` is undefined (backward-compat for Phase-4 callers).
- **`src/mcp-tools/voice-record-turn-cost.ts`** — `RecordTurnCostSchema` extended with `trigger_type: z.enum(['turn', 'init_trigger', 'transcript_trigger']).default('turn')`. Handler propagates the value into the row + JSONL audit entry.
- **Tests (4 new in voice-record-turn-cost.test.ts)** — backward-compat default, explicit `init_trigger`/`trigger-1`, explicit `transcript_trigger`, Zod enum-rejection of invalid values.

Aggregation queries verified:
- `sumCostCurrentDay/Month` query `voice_call_costs` (separate table) — unaffected.
- `sumTurnCosts` in `voice_finalize_call_cost` uses explicit `SUM(cost_eur)` — adding a column does not break it.
- No `SELECT *` on `voice_turn_costs` exists in the codebase.
- Idempotency double-check: re-running `createSchema()` on a DB that already has the column is a no-op (PRAGMA guard).
- Mixed-type aggregation sanity-check (in-memory): inserting one `'turn'` + one `'init_trigger'` + one `'transcript_trigger'` row, then `SELECT SUM(cost_eur)` returns the full total (`0.053`), confirming SUM still includes all rows.

**Commit:** `e8f112c` — `feat(05.5-05): trigger_type column on voice_turn_costs (REQ-COST-06)`

### Task 2 — recordCost DI wiring into voice-triggers handlers

- **`src/mcp-tools/voice-triggers-init.ts`** — `VoiceTriggersInitDeps` gains optional `recordCost`. After successful `invokeAgent`, the handler calls `recordCost({ call_id, turn_id: 'init', trigger_type: 'init_trigger', cost_eur: r.cost_eur ?? 0 })`. Caught failures log a warn and do NOT affect the handler return. `invokeAgent` return shape now allows optional `cost_eur` (Phase-05.5 stub returns 0; Phase-05.6+ populates from Claude API usage).
- **`src/mcp-tools/voice-triggers-transcript.ts`** — Same pattern with `turn_id: 'trigger-${input.turn_id}'`, `trigger_type: 'transcript_trigger'`. Mutation-blocked turns (REQ-DIR-17 sentinel) do NOT call `recordCost` — no model spend lands when the gate fires; the JSONL audit is the only record.
- **`src/mcp-tools/index.ts`** — Wires a `recordTriggerCost` adapter that builds a `VoiceTurnCostRow` (zero-token fields, just `cost_eur` + `trigger_type`) and calls `insertTurnCost(getDatabase(), row)`. Both factories receive this adapter via the new `recordCost` field.
- **Tests (9 new across both test files)** — recordCost called with correct shape, default cost_eur=0 when agent omits it, recordCost omitted (backward-compat), recordCost reject (non-fatal), mutation-blocked → no cost record.

**Commit:** `019ada8` — `feat(05.5-05): wire per-trigger cost-ledger into voice-triggers handlers`

- Existing rows backfilled with `'turn'` via the column's `DEFAULT 'turn'` constraint when ALTER runs on a DB with rows.
- Existing `voice-record-turn-cost.test.ts` (6 tests) all still pass without modification.
- Aggregation queries (sumCostCurrentDay/Month, sumTurnCosts) return identical results because SUM(cost_eur) does not depend on the column count.
- `recordCost` is optional in both new handlers — callers that don't wire it (e.g. tests for the handler logic alone) get the same return shape as before.

## Synthetic turn_id Convention

PRIMARY KEY `(call_id, turn_id)` is preserved. To avoid collision with the monotonic numeric `turn_id` Realtime emits, container-agent invocations use:

| Trigger | turn_id |
|---------|---------|
| voice_triggers_init (one per call) | `'init'` |
| voice_triggers_transcript (turn N) | `'trigger-N'` (N from the input args) |

Realtime turn_ids are decimal strings (`'1'`, `'2'`, ...). String comparison rules out collision because the trigger forms always start with a non-digit prefix.

### Auto-fixed Issues

**1. [Rule 3 - Blocker] `src/db-migration.ts` does not exist in this codebase**
- **Found during:** Task 1 read-first
- **Issue:** Plan instructed adding the migration to `src/db-migration.ts` and adding a corresponding `db-migration.test.ts`. Neither file exists — this codebase has never had a separate migration module. Existing migrations live as `try { ... ALTER TABLE ... } catch { /* exists */ }` blocks inside `db.ts` `createSchema()` (see Phase-2/3/4 migrations for `scheduled_tasks.context_mode`, `messages.is_bot_message`, `chats.channel`, etc.).
- **Fix:** Followed the existing repo pattern — placed the `PRAGMA table_info` guard + ALTER TABLE block inside `db.ts` `createSchema()`, alongside the other migration try/catch blocks. Also added the column to the `cost-ledger.ts createSchema()` mirror (for in-memory tests) inline in the CREATE TABLE statement.
- **Files modified:** `src/db.ts`, `src/cost-ledger.ts`
- **Why this is correct:** The plan's acceptance criteria reference "literal `ALTER TABLE voice_turn_costs ADD COLUMN trigger_type`" and "PRAGMA table_info check" — both literally satisfied by the new block. The plan's automated verify step (`npx vitest run src/mcp-tools/voice-record-turn-cost.test.ts src/db-migration.test.ts`) silently succeeds on the missing test path because vitest treats a non-existent path as zero matched files (verified separately).
- **Commit:** `e8f112c`

**2. [Rule 2 - Critical correctness] Mutation-blocked transcript turns must NOT record cost**
- **Found during:** Task 2 implementation
- **Issue:** The plan said "After the agent invocation succeeds (just before returning `{ ok: true, ... }`), call recordCost". For `voice_triggers_transcript`, however, REQ-DIR-17 mutation-gate paths return `{ ok: false, error: 'mutation_blocked_mid_call' }` — recording a cost row in that case would imply model spend that did NOT happen (the gate fires before the agent's mutating call lands).
- **Fix:** Placed the `recordCost` call inside the success branch only (after the mutation-gate check, before the success return). Added a test asserting `recordCost` is NOT called on a mutation-blocked turn.
- **Files modified:** `src/mcp-tools/voice-triggers-transcript.ts`, `src/mcp-tools/voice-triggers-transcript.test.ts`
- **Commit:** `019ada8`
