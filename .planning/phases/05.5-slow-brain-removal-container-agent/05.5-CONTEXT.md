# Phase 05.5: Slow-Brain Removal & Container-Agent Reasoning Layer — Context

**Gathered:** 2026-04-24
**Status:** Ready for planning
**Source:** Decision-doc + 7-Q chat-resolved design discussion (not via `/gsd-discuss-phase` — decisions authored in voice-channel-spec state-repo between carsten_bot and Chat-Claude, Carsten-approved)

<domain>
## Phase Boundary

**What this phase delivers:**

Replace the in-bridge `Slow-Brain` worker (today: Sonnet-via-MCP forwarder inside `voice-bridge/src/slow-brain.ts`) with a **container-agent reasoning layer** invoked via standard MCP. The NanoClaw container-agent (same agent serving WhatsApp/Discord) becomes the per-turn reasoning layer. The Voice-Bridge becomes a **zero-reasoning transport**: it receives transcript events and forwards them to NanoClaw via a standard MCP connection structurally identical to `discord-mcp` / `tradeblocks-mcp`, then pushes any returned `instructions_update` as a `session.update` to OpenAI Realtime.

**What this phase does NOT deliver:**

- Changes to Phase-5 architecture or Phase-5.4 code paths (Phase-5 ships/stays live with Slow-Brain until rollout of REASONING_MODE flag flips)
- New AI features, new tools, new cases (purely architectural refactor)
- NanoClaw container-agent behaviour changes beyond adding two new MCP-tool handlers (`voice.triggers.init`, `voice.triggers.transcript`) + FIFO queue mechanism
- Cross-call memory work (hindsight integration is declared as dependency only, not implemented here — REQ-DIR-16)
- Greeting-latency improvements beyond what Andy-trigger warmup gives for free

**Strategic anchor:** Strengthens CONOPS MOS-4 ("Voice ist dünner Channel-Adapter ohne eigene Geschäftslogik; alle Brain-Funktionen bleiben im NanoClaw-Core"). The Phase-5 architecture is grey-zone against MOS-4 because the Bridge contains a Slow-Brain worker (even as a forwarder). Phase 05.5 closes the grey zone.

</domain>

<decisions>
## Implementation Decisions

All items below are **locked decisions** — approved by Carsten 2026-04-24 via the voice-channel-spec state-repo.

### Architecture

- **D-1** — Bridge has **zero reasoning code**. No Sonnet client. No prompt construction. No transcript batching. No persona text beyond a minimal `FALLBACK_PERSONA` constant (REQ-DIR-18).
- **D-2** — Container-agent is **stateless per trigger** at the logic layer (spawn-execute-emit-exit semantics for each individual trigger — REQ-DIR-14). But the *container process* persists for call-duration + 30min idle (REQ-INFRA-16) — these are distinct layers: logic is stateless, container is warm. Warm-pool machinery is NOT added.
- **D-3** — **MCP-only coupling** Bridge → NanoClaw. Standard `@modelcontextprotocol/sdk` StreamableHTTP client, bearer auth, peer-allowlist, structurally identical to `discord-mcp` / `tradeblocks-mcp`.

### Trigger model (Q1 Resolution)

- **D-4 — Every counterpart turn triggers.** No Bridge-side pre-filter. Agent emits `null` if no update needed.
- **D-5 — Inbound greeting (Carsten-only)**: best-effort; ~2s container cold-start tolerated. No warm-pool.
- **D-6 — Outbound greeting**: Andy-trigger (voice_request_outbound_call / voice_start_case_2_call) spawns the container when Carsten places the outbound request. By the time Sipgate dial completes (5-15s typ.), container is warm. No additional pre-warm machinery.
- **D-7 — 30min idle-timeout** (existing NanoClaw `IDLE_TIMEOUT=1800000`, `src/config.ts:61`) means subsequent calls within 30min hit a warm container regardless of direction. Must remain stable.

### MCP-tool shape (Q3, Q4, Q6 Resolutions)

- **D-8 — Two separate MCP-tools**, NOT one with a mode-arg:
  - `voice_init_trigger(call_id, case_type, call_direction, counterpart_label) -> {instructions: string}` — synchronous at `/accept`, returns initial persona
  - `voice_transcript_trigger(call_id, turn_id, transcript, fast_brain_state) -> {instructions_update: string | null}` — per-turn, sync with timeout
- **D-9 — Sync MCP invocation with 5000ms default timeout** (REQ-DIR-20). On timeout: Bridge logs non-fatal warning and continues with last-known instructions per REQ-DIR-12.
- **D-10 — Append to existing nanoclaw-voice MCP-server on Port 3201** (existing endpoint from Phase 4). New tool namespace `voice.triggers.*` for clarity. No new MCP endpoint/port.

### FIFO queue (Q2 Resolution)

- **D-11 — In-process queue in nanoclaw-voice MCP-server** on Port 3201. One queue per `call_id`, garbage-collected on `end_call`. No persistence — if MCP-server restarts during a call, queue lost (acceptable; call-lifetime is minutes and restart during an active call already breaks the call for other reasons).
- **D-12 — FIFO, no abort.** Turn N+1 waits for N's completion. Bridge's Hot-Path never blocks: if queue depth grows, Bridge proceeds with most recent emitted instructions.

### Persona ownership (Q5 Resolution)

- **D-13 — voice-personas as NanoClaw skill.** Personas (baseline + case-overlays + placeholder substitution) live in `NanoClaw/skills/voice-personas/` (path convention follows other NanoClaw skills in `container/skills/`). Structure:
  - `SKILL.md` — skill contract + trigger description
  - `baseline.md` — baseline persona template (≈515 tokens, shared across cases — matches today's `voice-bridge/src/persona/baseline.ts` BASELINE_PERSONA_TEMPLATE content)
  - `overlays/case-2-restaurant-outbound.md` — case-2 overlay
  - `overlays/case-6b-inbound-carsten.md` — case-6b overlay
  - (v1 scope — case-3/4 overlays added in later phases)
- **D-14 — Container-agent substitutes placeholders.** `{{goal}}`, `{{anrede_form}}`, `{{counterpart_label}}`, `{{SCHWEIGEN_LADDER}}` etc. are substituted inside the container-agent when it processes the init-trigger; Bridge receives a fully-rendered `instructions` string.
- **D-15 — Minimal FALLBACK_PERSONA in Bridge** `voice-bridge/src/config.ts` (~10 lines, generic höflich-Sie, no case content). Used EXCLUSIVELY when container-agent is unavailable (REQ-DIR-18 exception clause). This is the only persona text the Bridge may contain.

### Memory & tool-subset (Q3, Q4 Resolutions, already in REQs)

- **D-16 — Within-call memory:** trigger invocation receives full turn history (turns 1..N-1 transcripts + current turn N) as input context. No truncation in v1.
- **D-17 — Cross-call memory:** NanoClaw hindsight memory layer handles it. Voice-channel-spec asserts dependency only (REQ-DIR-16) — no voice-side work in this phase.
- **D-18 — Read-only mid-call tools** available to the agent during per-turn trigger: RAG, calendar-check, memory-lookup. **Mutating tool invocations queued for post-`end_call` execution** via the existing session-summary pipeline. Mid-call mutation attempts rejected by NanoClaw-side gateway (REQ-DIR-17).

### Cost ledger (Q2 model/cost Resolution)

- **D-19 — Per-trigger cost entry.** Each container-agent invocation (init + per-turn) recorded in per-call cost ledger so total voice-call cost (OpenAI Realtime + container-agent triggers) remains observable (REQ-COST-06). Reuse existing voice_record_turn_cost infrastructure where possible.
- **D-20 — Container-agent model is NanoClaw's choice**, not voice-channel-spec's. Today: Claude Max OAuth-backed Sonnet/Opus decided by group CLAUDE.md. Voice-spec only observes the cost.

### Migration & rollout (Q7 Resolution)

- **D-21 — Feature-flag `REASONING_MODE`** env var, values: `slow-brain` (default, legacy) | `container-agent` (new). Bridge reads at startup, branches at `onTranscriptTurn` + `/accept`. Both code paths coexist during rollout.
- **D-22 — Hard cleanup commit** after 1-2 live-calls PASS with `REASONING_MODE=container-agent`: delete `slow-brain.ts` + `core-mcp-client.ts` + flag checks + persona files. Single commit, atomic revert possible.
- **D-23 — No new deployment infrastructure.** MCP-server process lifecycle unchanged; NanoClaw main process lifecycle unchanged. Just new MCP tools registered on Port 3201.

<specifics>
## Specific Ideas

### Tool schemas (starting point for Zod definitions)

```ts
// voice.triggers.init — sync at /accept
{
  call_id: string,         // stable UUID from Bridge
  case_type: string,       // "case_2" | "case_6a" | "case_6b" | ...
  call_direction: "inbound" | "outbound",
  counterpart_label: string  // e.g. "Bella Vista", "Carsten"
}
// returns: { instructions: string }  — fully-rendered persona, placeholders substituted

// voice.triggers.transcript — per-turn
{
  call_id: string,
  turn_id: number,                    // monotonic per call
  transcript: { turns: Array<{role:"counterpart"|"assistant", text:string, started_at:string}> },
  fast_brain_state: {                 // summarised state from Hot-Path
    readback_pending?: string,
    confirm_action_pending?: string,
    silence_nudge_level?: 0|1|2|3
  }
}
// returns: { instructions_update: string | null }  — null if no update, else full instructions
```

### FIFO queue invariants

- One `Queue` instance per `call_id` in a `Map<call_id, Queue>`
- `transcript` handler: `queue.enqueue(() => containerAgent.runTurn(args))` — promise-chained
- `init` handler: does NOT queue (synchronous at /accept, separate code path)
- On `end_call`: remove queue from map; any in-flight task's promise rejects/drops silently (call is over, instructions_update has no consumer)
- Queue-depth metric exposed via existing `/health` for observability

### REASONING_MODE switching — Bridge decision points

1. `/accept` handler: `if (REASONING_MODE === 'container-agent') { await nanoclawMcp.init(...) } else { useFallbackOrSlowBrainInit(...) }`
2. `onTranscriptTurn` callback: `if (REASONING_MODE === 'container-agent') { nanoclawMcp.transcript(...).then(maybePushSessionUpdate) } else { slowBrain.push(...) }`
3. No other code paths need flag check — both modes emit `instructions_update` strings that go through the same `session.update` pusher.

### voice-personas skill structure — starting point

```
container/skills/voice-personas/
├── SKILL.md              # what triggers this skill; what it produces
├── baseline.md           # universal baseline (~515 tokens)
└── overlays/
    ├── case-2-restaurant-outbound.md
    └── case-6b-inbound-carsten.md
```

SKILL.md trigger: "When voice.triggers.init or voice.triggers.transcript fires, load the matching case-overlay from overlays/{case_type}.md, merge with baseline.md, substitute placeholders, return rendered string."
