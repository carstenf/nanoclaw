**Error class hierarchy:**

```
Error

```

- `NanoclawMcpError` carries the server-emitted `error` string as `.code` (e.g. `'agent_unavailable'`, `'mutation_blocked_mid_call'`) so Bridge call sites can distinguish failure modes via `err.code` without string-matching.
- `NanoclawMcpTimeoutError` extends `NanoclawMcpError` so Bridge code can `instanceof NanoclawMcpError` for the broad fallback path (REQ-DIR-12) and `instanceof NanoclawMcpTimeoutError` for the specific timeout-metric path.
- Distinct from `CoreMcpError` / `CoreMcpTimeoutError` so the parallel rollout window (Phase 05.5 â†” 05.6) does not collide on `instanceof` checks.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 â€” Bug] ensureConnected() outside try block leaked TypeError**

- **Found during:** Task 1 verification (`tests/nanoclaw-mcp-client.test.ts > connection error`)
- **Issue:** Initial implementation copied `core-mcp-client.ts:124` verbatim â€” `await this.ensureConnected()` BEFORE the `try` block. Test `connection error â†’ throws NanoclawMcpError` got a raw `TypeError: fetch failed` from undici (since the unbound port :1 caused the underlying fetch to throw inside `transport.start()`), violating the public error-class contract that says "transport-level failures surface as NanoclawMcpError".
